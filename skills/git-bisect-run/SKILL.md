# Git Bisect Run (回帰コミットの特定)

回帰バグ（以前は動いていたのに、あるコミットから壊れた）の**原因コミットを `git bisect` で特定する**スキルです。成果物は「どのコミットが症状を持ち込んだか」の特定と、その根拠。**修正はこのスキルの範囲外**です（修正に進むなら再現テストを使って [tdd](../tdd/SKILL.md) の Red へ）。

## このスキルの核: 述語スクリプトの合成と、手動反復の禁止

このスキルの核は、**`git bisect run` に渡す述語スクリプト（そのコミットで症状が出るかを 0/非 0 で返すスクリプト）を合成すること**です。エージェントの仕事は「正しい述語を作ること」に集中させます。

そして絶対の規律として、**各ステップのテストをエージェントが手で行いません**。`git bisect good` / `git bisect bad` を手で繰り返し、途中のコミットを一つずつ自分で checkout して試すのは**禁止**です。反復は必ず `git bisect run <script>` に委ねます。

理由:

- **手動反復はトークンと時間を浪費する。** bisect は探索を対数回に抑える仕組みなのに、各ステップを人手で回すとその利点をエージェントのターン数で食い潰す。
- **手動反復は判定ブレを生む。** 同じ「症状が出るか」を、ステップごとに微妙に違う基準（違うコマンド・違う観察の仕方）で判定してしまい、途中で基準が揺れると bisect の結論が丸ごと狂う。**述語スクリプトへの固定こそが、全ステップで同一の判定基準を使うことを保証する。**

唯一許される手動実行は、本走の前に述語自体を検証する「4. 述語の校正」だけです。これは「各ステップを手で試す」こととは別物です。

## bug-repro との合成

述語の中身は「バグが再現するか」の判定であり、これは [bug-repro](../bug-repro/SKILL.md) の成果物そのものです。したがって:

- **再現テスト／再現スクリプトがまだ無ければ、先に [bug-repro](../bug-repro/SKILL.md) を実行して作る。** いきなり述語を書こうとしない。再現できるものが無ければ bisect は始められない。
- bug-repro の「下流との受け渡し規約」──**単一コマンドで実行でき、終了コード 0 / 非 0 で合否を判定できる**成果物──を受け取り、それを bisect の述語スクリプトで包む。bug-repro はまさにこの契約（`git bisect run` に述語として渡せること）を意識して成果物を整えている。
- **既にユーザーが再現コマンドを持っているなら、それをそのまま使う。** 改めて bug-repro を回す必要はない。ただしその 1 コマンドが「単一コマンド・終了コードで合否」を満たすことは確認する。

要は、bug-repro が「症状を 0/非 0 で表現できる 1 コマンド」を用意し、このスキルがそれを bisect が反復できる述語に仕立てる、という分業です。

## 全体の流れ

1. ガードと前提確認（git リポジトリ、bad / good の確定）
2. 再現手段の確保（bug-repro との合成、再現コマンドを 1 つに確定）
3. 述語スクリプトの合成（このスキルの核）
4. 述語の校正（唯一許される手動実行）
5. 実行（専用 worktree で `git bisect run`）
6. 結果の検証と報告
7. 後片付け

## 1. ガードと前提確認

- **git リポジトリであること**を確認する（`git rev-parse --is-inside-work-tree`）。そうでなければ対象外として終了する。
- 探索の両端を確定する:
  - **bad** ── 症状が出るコミット。通常は `HEAD`、または報告された版（タグ・SHA）。
  - **good** ── 症状が出ないと分かっているコミット。最後のリリースタグ、ユーザーの記憶にある「この頃は動いていた」版など。
- **good が不明なとき、候補を勝手に大量試行しない。** リリースタグ（`git tag`）や日付（`git log --until=<date>`）から**候補を挙げてユーザーに確認する**。良さそうな古いコミットを片端から試すのは、手動反復の禁止（このスキルの核）に反するうえ無駄が多い。
- good は bad の**祖先**でなければならない（`git merge-base --is-ancestor <good> <bad>`）。祖先関係が無ければ端の選定を見直す。

## 2. 再現手段の確保

「bug-repro との合成」の節に従い、**単一コマンドで実行でき終了コード 0/非 0 で合否が判定できる再現手段**を 1 つに確定します。この 1 コマンドが、次の述語スクリプトの中核になります。再現が確保できない（bug-repro でも再現しない）なら、bisect には進めません ── その旨を正直に報告します。

## 3. 述語スクリプトの合成 ── このスキルの核

### 終了コードの規約

`git bisect run` は述語スクリプトの終了コードを次のように解釈します。**この使い分けを述語の設計の起点にします。**

| 終了コード | bisect の意味 | 使いどころ |
|---|---|---|
| `0` | **good**（このコミットは症状が出ない） | 再現コマンドが「症状なし」を示した |
| `1`–`127`（`125` を除く） | **bad**（このコミットは症状が出る） | 再現コマンドが「症状あり」を示した |
| `125` | **skip**（このコミットは判定不能） | ビルド失敗・依存解決失敗など、**バグの有無と無関係**な理由で判定できない |
| `128` 以上（特に `255`） | **bisect 全体を中断** | 使わない。シグナル死などでここに落ちうるなら `1` に正規化する |

肝は **`125` = skip の使い分け**です。ビルドや依存解決の失敗は「バグがあるかどうか」とは無関係なので、それを bad と誤判定させないために `125` を返して skip します。skip されたコミットは bisect が近傍の別コミットで代替します。

### スクリプトの構造

述語スクリプトは次の 3 段構成にします。

1. **ビルド・依存準備。** 失敗したら `exit 125`（skip）。バグの有無と無関係な失敗をここで吸収する。
2. **再現コマンドの実行。** 「2. 再現手段の確保」で確定した 1 コマンドを走らせ、終了コードを捕まえる。
3. **意味への変換して `exit`。** 再現コマンドの終了コードを bisect の意味（good=0 / bad=非0）に変換する。

堅牢化として `set -u`（未定義変数で失敗）などを付けます。テストの終了コードが `128` 以上になりうる場合（シグナル死など）は、そのまま返すと bisect が中断してしまうため **`1` に正規化**します。骨組みの例:

```sh
#!/usr/bin/env bash
set -u

# (1) ビルド・依存準備。失敗はバグと無関係 → skip(125)
<build/install commands> || exit 125

# (2) 再現コマンドの実行（bug-repro の成果物 = 単一コマンド）
<repro command>
rc=$?

# (3) bisect の意味へ変換
if [ "$rc" -eq 0 ]; then
  exit 0            # 症状なし = good
elif [ "$rc" -ge 128 ]; then
  exit 1            # シグナル死などは bad に正規化（中断を避ける）
else
  exit 1            # 症状あり = bad
fi
```

> 再現コマンドの「終了コード 0 = 症状あり」なのか「非 0 = 症状あり」なのかは bug-repro の成果物次第です。bug-repro の規約では**症状が出たとき非 0**（＝失敗テスト）なので、上の例は「非 0 = bad」に対応しています。成果物の意味が逆なら (3) を反転させます。**どちらの向きかを取り違えると bisect の結論が丸ごと逆になる**ので、次の「4. 述語の校正」で必ず確かめます。

### スクリプトと再現テストは検査対象のワークツリーの外に置く

bisect は各ステップで**古いコミットを checkout する**ため、作業ツリー内に置いた述語スクリプトや再現テストは、checkout のたびに消えたり、過去の同名ファイルと衝突したりします。したがって:

- 述語スクリプトと（フレームワーク非依存の）再現スクリプトは、**bisect が checkout するワークツリーの外**に置く。リポジトリ外、またはセッションのスクラッチ領域が適所。
- `git bisect run` には**絶対パス**で渡す。相対パスは checkout で足元が変わると壊れる。

### 古いコミットで再現テストが動かない問題

テストフレームワークや依存ライブラリの版が bisect の探索範囲の中で変わっていると、**フレームワーク経由のテストは古いコミットで壊れます**（そのコミットのコードは新しいテストランナーを前提にしていない、など）。これは症状とは無関係な失敗であり、bad と誤判定させたくありません。対策:

- 範囲が短ければ、そういう失敗は述語の (1) で `exit 125`（skip）に落とす。
- **範囲が長い場合は、フレームワーク非依存の自己完結スクリプト**（bug-repro の「再現スクリプト」形式 ── ランナーに依存せず単体で実行でき、症状で非 0 終了する）に倒すことを推奨する。範囲全体で同じ 1 本のスクリプトが動くようにしておくと、判定基準が全ステップで一定に保たれる（＝このスキルの核と整合）。

### フレーキーな再現

bisect は**判定の決定性**を前提とします。同じコミットを 2 回測って違う答えが出ると、探索は破綻します。bug-repro が**再現率つき**の成果物（例: 「20 回中 3 回失敗」）を渡してきた場合は、そのままでは述語に使えません。

- 述語の (2) の中で**再現コマンドを N 回反復**し、「**1 回でも再現したら bad**」のように固めてから使う。
- **N は、その再現率で偽陰性（本当は bad なのに good と誤る）が十分小さくなるよう選ぶ。** 再現率 p なら、N 回中 1 回も出ない確率は概ね (1−p)^N。例えば p=0.15 なら N=20 で (0.85)^20 ≈ 0.04、N=30 で ≈ 0.008。許容する誤判定率から逆算して N を決める。反復ぶん実行時間は伸びるので、範囲の長さと相談する。

## 4. 述語の校正 ── 唯一許される手動実行

本走の前に、述語スクリプトを **bad コミットで 1 回・good コミットで 1 回**だけ実行し、期待どおりに返すことを確認します。

- **bad コミットで実行 → 非 0（bad）** を返すこと。
- **good コミットで実行 → 0（good）** を返すこと。

これは「各ステップを手で試す」ことではなく、**述語そのものの検証**です。ここが狂っていると bisect 全体が無意味になる（誤ったコミットを断定する）ため、必須の 1 手間です。校正が通らなければ:

- 述語を直す（終了コードの向きの取り違え、ビルド段の不足、パスの誤りなど）。
- それでも bad で bad・good で good にならないなら、**good/bad の選定ミス**を疑う（good が実は既に壊れている、bad が実は症状を出さない）。端を選び直す。

校正が両方通って初めて本走に進みます。

## 5. 実行

bisect は checkout を繰り返すので、**メインの checkout を占有せず、dirty state と混ざらない専用の worktree** で回します。[concurrent-dev](../concurrent-dev/SKILL.md) の規約に合わせ、プロジェクトルート直下の `.wt/` に作ります（`.wt/` はリポジトリを汚さないよう `.gitignore` ではなく `.git/info/exclude` に入れる）:

```sh
git check-ignore -q .wt || echo '.wt/' >> .git/info/exclude
git worktree add .wt/bisect <bad>
cd .wt/bisect
```

その worktree の中で bisect を開始し、run に**述語スクリプトの絶対パス**を渡します:

```sh
git bisect start <bad> <good> [-- <paths>]
git bisect run /abs/path/to/predicate.sh
```

- **`git bisect start <bad> <good>`** で両端を一度に与えます（`start` の第 1 引数が bad、第 2 引数が good）。
- **変更が特定のディレクトリに限られると分かっているなら `-- <paths>` で範囲を絞る。** そのパスに触れないコミットを探索から外せるので、ステップ数（=ビルド＋テストの回数）を減らせる。

あとは `git bisect run` が good/bad/skip を自動で辿り、`<sha> is the first bad commit` を出すまで反復します。**この反復に手を出さない**のがこのスキルの核です。

## 6. 結果の検証と報告

bisect が名指ししたコミットを**鵜呑みにしません**。フレーキーさや skip の連続で、bisect が間違ったコミットを指すことがあります。

- **`git bisect log` を保存する。** 辿った good/bad/skip の履歴。再走・監査に使える。
- **`git show --stat <first-bad>` と差分**で当該コミットの中身を見て、**そのコミットの変更が本当に症状を説明できるか**を確かめる。症状の出る箇所・条件と、変更されたコード・ロジックが噛み合っているか。
- 説明できるなら、根拠とともに断定する。**説明できない／怪しいなら、その旨を正直に報告する**（skip が多かった、フレーキーで N 回反復に頼った、範囲にマージが多かった、等の弱点も添える）。断定できないときに無理に断定しない。

**報告内容:**

- 原因コミット: **SHA・件名・作者・日付**。
- **そのコミットの何が症状を起こすか**の分析（差分に基づく）。
- **bisect log**（辿った経路）。
- 検証の弱点があればそれも（skip の多発・フレーキーさ・範囲の性質）。

**修正はこのスキルの範囲外**です。修正に進むなら、確保した再現テストを [tdd](../tdd/SKILL.md) の Red 入口として引き渡します。

## 7. 後片付け

- `git bisect reset` で bisect 状態を解除する（worktree を消す前に、その worktree 内で行う）。
- worktree を削除する: `git worktree remove .wt/bisect`（消し残りがあれば `git worktree prune`）。
- 述語スクリプト・一時的な再現スクリプトなど、**セッション中に作った一時ファイルを片付ける**。

## やらないこと

- **各ステップを手で試さない。** `git bisect good/bad` を手動で繰り返さない。反復は `git bisect run` に委ねる（このスキルの核）。校正（4.）の 2 回だけが例外。
- **good を勝手に大量試行しない。** 不明なら候補を挙げてユーザーに確認する。
- **結論を鵜呑みにしない。** 差分で症状を説明できるか確かめ、できなければ正直に報告する。
- **修正しない。** 原因の特定まで。修正は再現テストを持って [tdd](../tdd/SKILL.md) へ。
