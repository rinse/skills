# Git Archaeology

「なぜこのコードはこうなっているのか」に答えるスキルです。`git blame` から始めてコミット・PR・Issue へとリンクをたどり、対象のコードや設計判断がなぜそうなっているかを掘り起こします。

**このスキルの核は、掘った事実を時系列で並べることではなく、「この設計判断の理由」と「いま変更してよいかの判断材料」への合成です。** コミット・PR を年代順に羅列するだけの報告は失敗であり、ユーザーの問いに直接答える形にまとめ直してください。

**このスキルは読み取り専用です。** `checkout` や `revert` などリポジトリの状態を変更する操作は一切行いません。

## 全体の流れ

1. 問いの特定
2. ローカル調査（git のみで完結する層）
3. 文脈調査（GitHub がある場合の層）
4. 合成と報告
5. 任意の後処理（知見の保存を提案）

## 1. 問いの特定

まず何を掘るべきかを明確にします。

- **対象**: 特定の行・ブロック、関数・メソッド、ファイル全体、あるいは「この設計上の癖」（例: なぜこの API はこう非同期なのか）のどれか。ユーザーの発言から特定できなければ、対象のファイル・行範囲・関数名を確認する。
- **知りたいことの種類**: 「歴史そのもの（誰がいつ何のために）」を知りたいのか、「いま変えても安全か」を判断したいのかで、最後にまとめる報告の重心が変わる。後者であれば「その制約は今も有効か」の検討により力を割く。

対象と問いの種類を掴んだら調査に入ります。

## 2. ローカル調査（git のみで完結する層）

対象の性質に応じてコマンドを使い分けます。

| 対象 | コマンド |
|------|----------|
| 特定の行・ブロックの変遷 | `git log -L <start>,<end>:<file>` |
| 特定の関数・メソッドの変遷 | `git log -L :<funcname>:<file>` |
| あるコード片（文字列）の出現・消滅 | `git log -S '<string>' --oneline`（pickaxe） |
| ある正規表現パターンの出現・消滅 | `git log -G '<pattern>' --oneline` |
| rename・ファイル移動をまたいだ追跡 | `git log --follow -- <file>` |
| blame の起点を掴む | `git blame -w -C -C -C <file>` |

- `-S`（文字列の追加・削除そのものを検出）と `-G`（差分中にその正規表現にマッチする行があるコミットを検出）は似ているが挙動が異なる。単純な文字列一致なら `-S`、パターンで探したいなら `-G` を使う。
- `git blame -w -C -C -C` は空白変更を無視し（`-w`）、同一コミット内・親コミット内・他ファイルからの copy/move まで遡って検出する（`-C -C -C`）。これにより一括フォーマットや行移動に隠れた本当の変更者に迫れる。
- blame が一括フォーマット・大規模リネーム・自動生成などの**表層のコミット**に当たったら、そこで終わらせず `git blame <sha>^ -- <file>` でその一つ前の状態に対して blame を取り直し、実質的な変更が入ったコミットまで掘り直す。この掘り直しは対象行が表層コミットで消えるまで繰り返してよい。

ここまでで得られるのは「いつ・誰が・どのコミットメッセージで変えたか」という一次情報です。コミットメッセージやコミット単体の diff だけで理由が十分に分かる場合は、次のステップ（GitHub 層）を省略してよい。

## 3. 文脈調査（GitHub がある場合の層）

コミットメッセージだけでは意図が分からない場合、その先の PR・Issue の議論まで掘ります。

### ツール検出

- `command -v gh` で `gh` の有無を確認する。**あれば `gh` を優先して使う。**
- `gh` が無い場合、GitHub MCP が利用できればそれで同じ調査を行う（`search_commits` / `search_pull_requests` / `pull_request_read` / `issue_read` など）。
- どちらも無い場合は**この層をスキップし、ローカル調査（手順 2）で分かったことだけを根拠に手順 4 の報告を作る**。ここで安易に「調べようがない」と投げ出すのではなく、コミットメッセージや diff から読み取れる範囲で理由を合成する。

「`gh` も GitHub MCP も無い」＝手段が無いだけであり、「そもそも GitHub 上のリポジトリではない」＝この層が対象外、という区別は混同しない。後者（GitHub リポジトリでない、ローカル限定の履歴等）の場合も同様にこの層をスキップしてよい。

### コミットから PR、PR から Issue へ

1. **コミットから PR を引く**
   ```
   gh api repos/{owner}/{repo}/commits/<sha>/pulls
   ```
   squash merge でコミットが 1 つに潰されていて上記で見つからない場合は、コミットメッセージ中の PR 番号（`(#123)` 等）や件名で探す。
   ```
   gh pr list --search <sha> --state all
   gh pr list --search "<コミット件名の一部>" --state all
   ```
2. **PR の説明・レビュー議論を読む**。なぜその実装を選んだか、代替案が却下された経緯、レビューでの懸念点などを拾う。
3. **リンクされた Issue をたどる**。PR の本文・コメントにある `#123` や `Closes #123` から Issue を開き、要件・制約の背景を確認する。

GitHub MCP を使う場合も、上と同じ順序（コミット → PR → Issue）で `search_commits` → `pull_request_read` → `issue_read` 等に読み替えて進める。

### 掘るのをやめる条件（コスト上限）

1 つの問いにつき、コミット → PR → Issue とたどるホップは**原則 3 段まで**とする。理由がその手前で判明したら、そこで打ち切ってよい（無理に 3 段目まで掘る必要はない）。

次のような「トレイルが冷えた」兆候に行き当たったら、それ以上は追わず正直に打ち切る。

- squash merge でコミットと PR のリンクが切れており、件名検索でも該当 PR が見つからない
- リポジトリ移行時の import コミット（大量ファイルを一括で追加する等）に行き着いた
- コミットメッセージ自体が無内容（"update", "fix" のみ等）で、PR も Issue も存在しない

この場合、**「ここから先は追えない」と正直に報告し**、そこまでに分かった事実と、そこから先の推測を明確に区別して手順 4 に進む。存在しない経緯をでっち上げてはならない。

### 大きい PR・差分の扱い

読むべき PR の差分やレビューコメントが大きくトークンを消費しすぎる場合、**要点抽出だけを Haiku の Subagent に委譲してよい**（`gh-release` と同じ委譲原則）。Subagent には「この PR/差分から、設計判断・却下された代替案・レビューで指摘された懸念を事実として抽出して報告する」ことだけを求め、**理由の解釈や結論の判断はさせない**。抽出結果を受け取ったら、それを自分（オーケストレーター）がローカル調査の結果と突き合わせて手順 4 の合成を行う。

## 4. 合成と報告

集めた事実を、以下の 4 つの見出しでまとめて報告します。単なる時系列ダンプで終わらせないこと。

- **経緯**: 要点のみの時系列（いつ・誰が・どのコミット/PR で）。細部を全部書き出さず、判断に関わる転換点だけを拾う。
- **理由**: 当時の制約・意図（PR の説明、Issue の要件、レビューでのやり取りなどから読み取れるもの）。
- **その制約は今も有効か**: 当時の前提（依存ライブラリのバージョン、要件、他コードとの依存関係など）がいま現在も成立しているかを検討する。成立しているかどうかは自明でないことが多いので、コードベースの現状（依存関係、呼び出し箇所など）を確認した上で判断する。
- **変更するなら注意すべきこと**: 制約が失効していても他に理由がある可能性、関連箇所への影響、テストの有無など、変更時に踏むべき手順や見落としやすい落とし穴。

**推測と事実は明確に区別して書く。** 「PR #123 の説明にこう書かれている」は事実、「おそらく〜という意図だったと考えられる」は推測、とラベルを分けて示す。手順 3 で「トレイルが冷えた」場合は、その旨をここで明記し、分かった範囲と推測の境界をはっきりさせる。

## 5. 任意の後処理

今回の調査で得られた知見（設計判断の理由、繰り返し参照されそうな経緯）が今後も再利用される価値を持つと判断した場合は、そのまま流さずに [okf-knowledge](../okf-knowledge/SKILL.md) を使って knowledge bundle に残すことをユーザーに提案する。ユーザーが希望すれば `okf-knowledge` スキルに引き継ぐ。一過性の調査で再利用価値が乏しい場合は提案しなくてよい。
