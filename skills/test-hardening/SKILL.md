# Test Hardening

既存コードのテストスイートを、リスクの高い場所から順に、質を守りながら増強するスキルです。テストの量産自体は安いモデルに委譲できますが、質が守られなければ増えるほど負債が積み上がります。そこで質の門番役をオーケストレーター（Fable 5 / Opus 級の高等モデル）に固定します。

## このスキルの核

**このスキルの核は 2 つの判断です。** (1) **リスクによる標的選定** ── カバレッジ % を目標にせず、壊れたとき痛い場所から埋めます。(2) **change-detector テストの排除** ── 現行実装をなぞるだけのテスト（バグも含めた現状をそのまま正解として固定するテスト）を弾く品質判定です。どちらも `coverage` コマンドの出力だけでは埋まりません。

分担の原則はこうです。**期待される振る舞いの仕様を書くのはオーケストレーター、それをテストコードに落とすのが Subagent。** コードだけを渡してテストを書かせると、Subagent は現行実装の挙動を「正しい仕様」とみなして追認するテストを書きます。バグまで固定してしまう change-detector が生まれるのはこの経路からです。仕様をオーケストレーターが先に書き、コードとの食い違いを自分の目でチェックしておくことで、この経路を断ちます。

## 全体の流れ

1. 前提確認
2. 標的選定
3. 振る舞い仕様の作成（オーケストレーター自身）
4. 委譲
5. 品質ゲート（オーケストレーターのレビュー）
6. マージ・コミット
7. 報告

## 1. 前提確認

- git リポジトリであり、作業ツリーが綺麗であること。未コミット変更があれば、先に退避するようユーザーに確認します。
- 既存のテストスイートが green であること。赤いまま標的選定を進めても、新しい失敗と既存の失敗を区別できません。
- テスト実行コマンドとカバレッジ計測コマンドを特定します。
- **mutation testing ツール（Stryker、mutmut、cargo-mutants、PIT 等）が導入済みなら、その結果を標的選定の一次情報として使います。** mutation score が低いファイルは、カバレッジ上は埋まっていても実質的に守られていない箇所だからです。未導入でも導入を必須にはしません。後段（5. 品質ゲート）のスポットチェックで代替できます。

## 2. 標的選定

カバレッジの穴を列挙したうえで、リスクでランク付けします。材料にするのは次です。

- **変更頻度** ── `git log` でよく触られているファイル。
- **過去のバグ修正の集中** ── fix コミットが繰り返し触るファイル。
- **公開 API・ドメインの中核であること。**
- **分岐の密度・複雑さ。**

逆方向の判断も明記します。**自明なコード（getter、定型的な glue）はカバレッジが低くても埋めません。** テストを書いて維持するコストに、その箇所が壊れたときの実害が見合わないからです。カバレッジ % を目標に据えると、この種の数字合わせテストを量産してしまいます。標的選定の軸はあくまでリスクであり、カバレッジ % そのものではありません。

標的リストと作業量の目安（どこを・どの深さまで埋めるか）をユーザーに提示し、着手前に確認を取ります。

## 3. 振る舞い仕様の作成（オーケストレーター自身）

各標的について、**「このコードが約束していること」** を箇条書きの仕様にします。この工程はオーケストレーター自身が行い、Subagent には渡しません。コードだけからテストを書かせると、バグも含めた現状を正解として固定する change-detector になってしまう、というのが理由です。

仕様の材料にするのは次です。

- ドキュメント・コメント。
- 型シグネチャ。
- 呼び出し側が置いている期待（実際の呼び出しコードから読み取る）。
- コミット履歴（その振る舞いが導入された経緯・意図）。

仕様を書く過程で、現行実装が約束と食い違っている疑いに気づくことがあります。その場で直そうとせず、バグ候補としてメモしておきます。判断はまだ確定させず、7. 報告で扱います。

## 4. 委譲

標的ごとに Subagent へ委譲します。並列実行の下回り（worktree の作成・レビュー・マージ・後片付け）は concurrent-dev のパイプラインに乗せます。

- **モデル選択** ── 基本は Sonnet。仕様が表駆動（入力と期待値の組を列挙した表）まで固まっていて、判断の余地がないものだけ Haiku に落とせます。
- テスト追加は、Subagent が自分でテストを実行して green を確認できる、数少ない自己検証可能なタスクです。委譲先にこの検証まで含めて任せられます。

Subagent への指示には次を含めます。

- **3. で書いた振る舞い仕様。**
- **倣うべき既存テストの場所** ── フィクスチャ・ヘルパ・命名の慣習がどこにあるか。
- **新しいヘルパを勝手に作らないこと。** まず既存のものを探させます。
- **書いたテストが現行コードで落ちる場合、テストを現行の挙動に合わせて曲げず、失敗するテストのまま報告すること。** それはバグ発見かもしれません。bug-repro が再現テストを「でっち上げない」のと同じ規律を、ここでは「テストを通すために曲げない」という形で課します。

## 5. 品質ゲート（オーケストレーターのレビュー）

マージ前に全テストをオーケストレーター自身が読みます。次に該当するテストは弾きます。

- **assertion が実質的にないテスト。**
- **mock-echo** ── モックに仕込んだ応答をそのまま検証しているだけのテスト。たとえば `mock.returns(42)` と設定したうえで `expect(result).toBe(42)` と確かめるだけのテストは、モックの設定を読み返しているにすぎず、対象コードの振る舞いを何も検証していません。
- **スナップショットの丸呑み** ── 出力を一度取ってそのまま正解にしただけのもの。
- **並列に書かれたテスト同士のフィクスチャ・命名の一貫性の欠如。** 全差分を一人で読むオーケストレーターにしか見えない観点です。

**mutation スポットチェック** も、この工程の柱です。標的コードに意図的なバグを 1〜2 個仕込み、新しいテストが落ちることを確かめてから、必ず元に戻します。落ちなければ、そのテストは実質的に何も検証していないので書き直しです。mutation ツールが導入済みなら、この確認をツールに委ねて構いません。

## 6. マージ・コミット

concurrent-dev の規律に従います。squash merge・一行メッセージでベースブランチに取り込み、タスク単位で検証と後片付けまで完結させます。

## 7. 報告

作業の最後に、次を報告します。

- 標的別の結果（追加したテスト・見送った箇所とその理由）。
- mutation チェックの結果。
- **テストが炙り出したバグ候補。** 直さずに報告することが前提です。修正は別のタスク（tdd や個別の修正作業）に委ねます。ここで直してしまうと、テスト追加の作業とバグ修正の作業が同じコミットに混ざり、どちらの判断も検証しづらくなります。

## やらないこと

- **カバレッジ % を目標にしません。** 数字合わせの自明なテストを量産する誘因になるためです。
- **現行コードで落ちるテストを、通るように書き換えません。** それはバグを隠すことと同じです。
- **新機能の開発には使いません。** そこは tdd の領分です。
- **単一のバグの再現には使いません。** そこは bug-repro の領分です。
