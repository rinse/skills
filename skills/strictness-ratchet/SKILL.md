# Strictness Ratchet

コンパイラ・リンタ・型チェッカの設定を今より厳しくし、噴出する診断を計画的に焼き払い、締めた設定を二度と緩めないためのスキルです。対象例は TypeScript の `strict`、mypy の `--strict`、ESLint のルール強化、clippy、`-Wall` などです。オーケストレーター（Fable 5 / Opus 級の高等モデル）での利用を想定しています。

## このスキルの核

**2 つの判断**がこのスキルの核です。機械的なコマンドでは埋まらず、オーケストレーターが握り続けます。

1. **トリアージ** ── 数百件規模で噴出する診断を、診断コード（ルール ID）× 修正パターンでバケットに分け、バケットごとに処理方針と担当モデルを決めること。
2. **fix / suppress / 報告の三方向判断** ── なかでも、**strict 化が炙り出した本物のバグを静かに直さない**こと。挙動を保存する修正だけを黙って進めてよく、挙動が変わる修正はバグ報告としてユーザーに上げます。

修正作業の大半は機械的で安いモデルに委譲できますが、この 2 つの判断はオーケストレーターの手元に残します。

## 全体の流れ

1. 前提確認
2. 目標の設定
3. 洪水の観測とバケット分け
4. トリアージ
5. バケット単位の実行
6. ラチェットを締める
7. 報告

## 1. 前提確認

- git リポジトリであり、作業ツリーが綺麗であること。未コミット変更があれば、先に退避するようユーザーに確認します。
- test コマンドを特定しておきます（バケットごとの検証に使います）。
- **現在の診断がゼロ（green）であること**を確認します。green でないなら、既存分を先に片付けるか、スコープをどう切るかをユーザーに確認します。既存の赤と strict 化起因の赤が混ざると、後段のバケット分けも件数の追跡も成立しません。

## 2. 目標の設定

どの設定をどこまで厳しくするかを決めます。**最終目標を一気に有効化せず、サブフラグ単位で段階的に進めます。** 例えば TypeScript の `strict` は `noImplicitAny` → `strictNullChecks` → 残りのフラグ、のように分解できます。mypy の `--strict` も同様に個別フラグへ分割できます。

各段階で有効化した直後に診断件数を観測してから、その段階の計画を立てます。先の段階まで見越して一括で計画しない理由は、後段のフラグが炙り出す診断が前段の修正内容に依存するためです。

## 3. 洪水の観測とバケット分け

対象フラグを（まだコミットせずに）有効にして、全診断を採取します。診断コードごとに集計し、**同じ診断コードでも修正パターンが異なるものはバケットを分けます。** 例えば TypeScript の `TS2532`（Object is possibly undefined）でも、オプショナルチェイニングの追加で閉じるものと、呼び出し元の型設計を見直す必要があるものは別バケットにします。

観測結果はバケット × 件数の表にまとめます。

| バケット | 診断コード | 修正パターン | 件数 |
|---|---|---|---|
| 例: optional-access | TS2532 | `?.` の追加 | 120 |
| 例: null-branch | TS2531 | null 分岐の追加（意味論的） | 34 |
| 例: unused-import | no-unused-vars | 未使用 import の削除 | 58 |

## 4. トリアージ

バケット単位で fix / suppress / 報告の三方向判断を行い、結果をユーザーに提示して確認を取ります。

- **fix（機械的）→ Haiku**: 修正が一意に定まるもの。明示的な型注釈の追加、未使用 import の削除など。
- **fix（意味論的）→ Sonnet またはオーケストレーター**: コードの意味を読まないと直せないもの。null 分岐の追加、型設計に触れる修正など。
- **報告 → ユーザー**: 診断が本物のバグを示唆するもの。`strictNullChecks` が見つけた実際に null になり得る参照、到達不能コードなど。**挙動を変える修正は勝手に行いません。**
- **suppress は最後の手段**とします。行単位の抑制コメントに限り、必ず理由を添えます。ルール全体・ファイル全体の無効化はしません。件数を消しただけになり、ラチェットが崩れるためです。

提示する表はバケット × 方針 × モデル × 件数の形にします。

| バケット | 方針 | モデル | 件数 |
|---|---|---|---|
| optional-access | fix（機械的） | Haiku | 120 |
| null-branch | fix（意味論的） | Sonnet | 34 |
| dead-code-suspect | 報告 | ユーザー | 3 |

## 5. バケット単位の実行

- **弱いモデルへの指示は修正パターンを before/after の模範例で固定します。** そのうえで「パターンに当てはまらない箇所は触らず報告する」という出口を必ず与えます。例外処理を弱いモデルに判断させないためです。
- **並列化の条件**: バケットが触るファイル集合が互いに素なら並列可、重なるなら直列にします。Subagent には git 操作をさせず、検証とコミットはオーケストレーターが行います。
- バケットごとのループは次の順です。
  1. 修正を委譲する。
  2. 対象診断の残数が計画どおり減ったことを確認する。
  3. テストが green であることを確認する。
  4. コミットする。1 バケット = 1 コミットとし、git-commit スキルの規約（一行メッセージ・ファイル明示列挙・`git add -A` 禁止）に従います。

## 6. ラチェットを締める

全バケットが完了したら、最後に設定ファイルの変更をコミットします。**修正が先、設定が最後**という順序を守ります。逆にすると履歴の途中状態が赤くなり、bisect を壊します。CI があるなら、その設定が CI でも強制されることを確認します。

ゼロにできなかったフラグは有効化を見送ります。進捗（残バケットと件数）は報告に残しますが、**半端に有効化して suppress で埋めることはしません。** それは診断を隠しただけで、ラチェットの意味を失わせます。

## 7. 報告

作業の最後に、次を報告します。

- 前後の診断件数（フラグ単位、バケット単位）。
- バケット別の処理結果（fix / suppress / 報告のどれで、何件処理したか）。
- **発見した本物のバグ**。報告のなかで最も重要な項目として強調します。
- suppress した箇所と、それぞれに添えた理由の一覧。

## やらないこと

- **push・PR 作成はしない。** 続けるなら gh-pr-create へ渡します。
- **ルール全体・ファイル全体の無効化で件数を消さない。**
- **バグ疑いを、挙動の変更込みで勝手に直さない。** 三方向判断のうち「報告」に振り分け、ユーザーの判断を待ちます。
