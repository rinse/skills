# Doc Sync

コード変更後、README やドキュメントが実体とずれていないかを検査し、**ずれた箇所だけ**を現在のコードから合成し直すスキルです。[gh-pr-update](../gh-pr-update/SKILL.md) が「PR の説明と差分のずれ」を直すのと同じ思想を、リポジトリのドキュメントに適用したものです。

このスキルの核は、**「何を直すか」より「何を直さないか」**です。エージェントはドキュメント全体を書き換えたがりますが、このスキルの規律は**実体とずれた箇所だけに手を入れ、文体・構成・書き手の声を保存する**ことにあります。直す箇所は追記でパッチせず、その節を現在のコードから合成し直します（合成主義）。「もっと良い書き方がある」はこのスキルの仕事ではありません。事実として実体とずれている箇所だけを直します。

## 全体の流れ

1. スコープの決定
2. 主張の棚卸し
3. ずれの報告と確認
4. 修正
5. コミットと報告

## 1. スコープの決定

既定のスコープは**「最近の変更が影響しうるドキュメント」**です。

- ブランチ作業中なら、ベースブランチとの差分（`git diff <base>...HEAD`、`git --no-pager log <base>..HEAD`）から特定します。
- そうでなければ、直近の変更（`git status`、直近のコミット）から特定します。
- **ユーザーがドキュメントを指定した場合はそれに従います。**

対象は**散文ドキュメント**（README、`docs/` 配下など）です。以下は対象外です:

- **CHANGELOG**（変更履歴は過去の記録であり、実体と照合して直すものではない）
- **コード内コメント**（このスキルはドキュメントファイルのみを扱う）

## 2. 主張の棚卸し

対象ドキュメントから、**コードの実体に紐づく記述**（＝実行すれば真偽が確かめられる主張）を抽出します。デザイン思想の説明や意図の記述など、実体と 1:1 対応しないものはここでは棚卸ししません（詳しくは「触ってはいけないもの」参照）。

主張の種類ごとに検証方法を変えます。

| 主張の種類 | 検証方法 |
|---|---|
| コマンド・CLI オプション | 実行が安全（読み取り専用・副作用なし）なら実行して確認する。副作用があるなら実装や `--help` 出力を読んで確認する |
| API シグネチャ・型・設定キー・既定値 | ソースコードを読んで照合する |
| ファイルパス・ディレクトリ構成 | 存在確認する（`ls` / glob） |
| コード例 | 副作用が無ければ実行して確認する。実行できない（外部サービスへの書き込み等）なら、現在の API と突き合わせて机上で検証する |

「実行が安全」の判断基準は、**読み取り専用で状態を変えないこと**です。外部サービスへの書き込みやローカル状態の変更を伴う例は、実行せず机上検証にとどめます。

## 3. ずれの報告と確認

見つかったずれを一覧にし、**修正前にユーザーへ提示して確認を取ります。** 一覧は次の形にします。

- **ドキュメントの記述**
- **実体**（ソース・実行結果などの根拠）
- **提案する修正**

ここで重要な判断基準があります。**「ドキュメントが古い」のか「コードがバグっている」のかは自明ではありません。** ドキュメントが本来の仕様を表していて、コードの方がそれを裏切っている可能性がある場合は、勝手にドキュメントを実装に合わせて書き換えないでください。そのようなケースは**「ずれ」として提示し、どちらを正とするかユーザーの判断を仰ぎます。**

ユーザーが承認したずれだけを次の修正に進めます。修正指示や却下があれば一覧を更新して再提示します。

## 4. 修正

**承認されたずれのみ**を修正します。

- ずれた節は、**現在のコードを根拠に書き直します。** 古い記述に追記してつじつまを合わせるのではなく、その節を実体から合成し直します（合成主義）。
- 周辺の**文体・見出し構成・言語を保存**します。日本語で書かれたドキュメントは日本語のまま直し、英語の文書に日本語を混ぜたり、その逆をしたりしません。
- 修正はステップ 3 で承認された箇所に限ります。ついでに目についた表現の改善や再構成には手を出しません。

### 触ってはいけないもの

以下は「ずれ」であっても、このスキルの対象として直しません。事実ベースの検証だけがこのスキルの仕事で、それ以外は書き手や別の判断に委ねます。

- **文体・トーン・構成** — 読みやすさの改善はこのスキルの仕事ではありません。
- **TODO・ロードマップなど、意図的に未来を書いた節** — 「まだ実装されていない」ことは実体とのずれではありません。
- **設計思想の説明** — 実装と 1:1 対応しない記述（「なぜこう設計したか」等）は、実行して真偽を確かめられないため対象外です。
- **バッジ・ライセンス表記** — ビルドステータス等の外部生成物や法的表記は、このスキルの検証対象ではありません。

## 5. コミットと報告

修正が承認された範囲に収まったら、[git-commit](../git-commit/SKILL.md) の規約（一行メッセージ・ファイル明示列挙、`git add -A` 禁止）に従ってコミットします。

例:

```
docs: sync README options table with current CLI flags
```

最後に次を報告します。

- **直した箇所**（ドキュメントの記述 → 実体 → 修正内容）
- **あえて直さなかった箇所** — ずれとして検出したが「ずれではない」と判断したもの、またはユーザーが却下したもの。理由も添えます。
