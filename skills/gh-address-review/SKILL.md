# GitHub PR Review への対応

自分の PR に付いたレビュー指摘を捌くためのスキルです。レビューする側（[gh-review](../gh-review/SKILL.md)）と、対応後の本文更新（[gh-pr-update](../gh-pr-update/SKILL.md)）の間にある **「レビューを受ける側」のワークフロー** を埋めます。

**このスキルは GitHub の PR 専用です。** GitHub の PR でない場合（GitHub リポジトリでない、GitLab 等の別ホスト、ローカル変更のみ等）は、このスキルの対象外なので何もしません。

## このスキルの核

核は、レビューコメントの **分類判断** です。「言われたから全部直す」のではなく、各指摘を読んで次の 4 つに振り分けます。

- **(a) 対応する** — 指摘が妥当。コードを直す。
- **(b) 押し返す** — 設計意図・トレードオフに照らして従うべきでない。根拠を添えて返信する。
- **(c) 質問で返す** — 指摘の意図・前提が不明瞭。確認の問いを返す。
- **(d) 対応不要と説明する** — 誤解・既に別箇所で対応済み・仕様通り。理由を添えて説明する。

**設計意図と衝突する指摘に盲従しないこと**がこのスキルの価値です。機械的な修正作業ではなく、この分類こそが判断の所在です。

## 全体の流れ

1. ガードと前提確認
2. コンテキストの収集（レビューコメント・差分・レビュー以降のコミット）
3. 分類 —（a〜d）に振り分け、**分類表をユーザーに提示して確認を取る**
4. 実装 —（a）を修正しコミットする
5. 返信の合成と確認 —（b）（c）（d）への返信文を組み立て、**投稿前に確認を取る**
6. 実行 — push・返信の投稿・スレッドの resolve
7. 後処理 — [gh-pr-update](../gh-pr-update/SKILL.md) で本文を最新化する

## 1. ガードと前提確認

- **`gh` コマンドが無い場合**（`command -v gh` で確認）→ このスキルは `gh` に依存する。未インストールならその旨を伝え、**インストールするかどうかユーザーに尋ねる**。希望する場合は実行環境（OS / パッケージマネージャ）に合った方法でインストールして続行する（公式は https://cli.github.com ）。希望しない場合はここで終了する。「`gh` が無い」ことと「GitHub PR でない」ことは別物なので混同しない。
- **そもそも GitHub PR が対象でない場合**（`gh` がリポジトリを認識しない／別ホストのリモート等）→ このスキルの対象外。その旨を伝えて終了する。

### 対象 PR の特定

対応する PR を一つに確定させます。誤った PR を触ると取り消しづらいので、次のいずれかで **明確な場合に限り** 進めます。

- **ユーザーが PR 番号や URL を共有している。**
- **文脈から対象が一意に明らか。** 例: 現在のブランチに紐づく PR がちょうど一つで、その PR の話をしている。

どちらでもなく対象が定まらない場合は、推測で進めず、どの PR か（番号・URL）をユーザーに尋ねます。

```
gh pr view --json number,title,headRefName,baseRefName,url
```

以降、対象 PR の番号を `<num>`、base ブランチを `<base>`、PR の作成元ブランチを `<branch>` とします。

## 2. コンテキストの収集

正しく分類・返信するには、「何を指摘されたか」だけでなく「その指摘がまだ有効か（レビュー後に自分が直していないか）」まで押さえる必要があります。

### レビューコメント（インラインスレッド含む）

- レビュー全体・要約コメント: `gh pr view <num> --json reviews,comments`
- インライン（行に紐づく）レビューコメント: `gh api repos/{owner}/{repo}/pulls/<num>/comments`（`{owner}`/`{repo}` は `gh` がカレントリポジトリから補完する）
- **スレッドと resolve 状態**は REST では取りにくいので GraphQL で引く。resolve するときにも thread の ID が要るので、ここで併せて取得しておく。

```
gh api graphql -f query='
{
  repository(owner: "{owner}", name: "{repo}") {
    pullRequest(number: <num>) {
      reviewThreads(first: 100) {
        nodes {
          id
          isResolved
          isOutdated
          path
          comments(first: 50) { nodes { author { login } body } }
        }
      }
    }
  }
}'
```

同じ行への往復（相手の再指摘・こちらの過去の返信）はスレッド単位で読み、会話の文脈を取り違えないようにします。`isOutdated` が付いたスレッドは、指摘対象の行が既に変わっている可能性を示します。

### 差分とレビュー以降のコミット

- 現在の差分: 対象ブランチをチェックアウト済みなら `git -P diff <base>...HEAD`、そうでなければ `gh pr diff <num>`。
- レビュー以降に自分が追加したコミット: `git --no-pager log <base>..HEAD`。**レビュー時点から差分が動いていれば、既に対応済み・状況が変わった指摘があり得ます**（それは (d) に振り分ける材料）。

## 3. 分類（このスキルの中心）

各指摘を (a)〜(d) に振り分けます。判断基準:

- **severity タグとの連動**（[gh-review](../gh-review/SKILL.md) のタグ体系）:
  - `[対応必須]` は「直す」（a）推定を **強く** 働かせる。ただし設計上の明確な根拠があれば押し返してよい。その場合は **必ず根拠 + 相手への問い** を添える（一方的な却下にしない）。
  - `[対応任意]` は著者判断に委ねられた提案。採否をここで決める。
  - タグなし（nit・疑問・質問）は、質問なら (c)、軽微な指摘なら (a) か (d)。
- **既に対応済み・仕様通り・誤解** に見えるものは (d)。レビュー以降のコミットで解決していないか 2 の情報と突き合わせる。
- **意図が読み取れない指摘** は勝手に解釈して直さず (c) で意図を問う。
- **複数レビュアーの指摘が矛盾する場合**は、勝手にどちらかを採らない。矛盾している事実をユーザーに提示し、どちらを採るか判断を仰ぐ。

### 分類表を提示して確認を取る

どれを直しどれを押し返すかは **ユーザーの意思決定** に関わります。実装・返信に進む前に、分類表を提示して了承を取ります。

| # | 指摘（ファイル:行 / 要約） | タグ | 分類 | 対応方針 |
|---|---|---|---|---|
| 1 | src/auth.py:42 null チェック漏れ | [対応必須] | (a) 対応 | ガード節を追加 |
| 2 | src/util.py:10 map に書き換えては | [対応任意] | (b) 押し返す | 既存の分岐は例外処理込みで map 化すると読みにくい旨を返信 |
| 3 | 命名を変えたら？ | なし | (c) 質問 | どの命名規約を想定しているか問う |

ユーザーが分類を修正したら反映して再提示します。ここで合意した分類が以降の作業の土台になります。

## 4. 実装（a の修正）

(a) に振り分けた指摘を修正します。

- コミットは [git-commit](../git-commit/SKILL.md) の規約に従う: **一行メッセージ**、conventional-commit 風 prefix、**対象ファイルを明示列挙**（`git add -A` は禁止、`git add -u` は可）。
- 論理的にまとまる単位でコミットする。指摘ごとに機械的に 1 コミットにする必要はないが、無関係な変更を一つのコミットに混ぜない。
- **修正はコミットまで済ませる。** 作業ツリーに未コミットの変更が残っていると、後段の push にも [gh-pr-update](../gh-pr-update/SKILL.md) の差分にも乗らない。

## 5. 返信の合成と確認

(b)(c)(d) に振り分けた指摘への返信文を組み立てます。

### 返信は乱発しない

**修正で応えた指摘（a）には、原則コードで語らせます。** GitHub 上の返信は控えめに、が原則です。返信を書くのは **反論（b）・質問（c）・説明（d）** が必要なものに絞ります。「直しました」だけの返信は基本的に不要で、6 の resolve が対応の事実を示します（一言添えたい場合のみ短く）。

### 返信文の指針

- **(b) 押し返す**: 設計上の根拠を具体的に述べ、**相手への問い**を添えて対話にする。「間違っている」ではなく「この意図でこうしたが、その前提だと問題があるか？」の形。
- **(c) 質問で返す**: 何が不明で、何を答えてほしいかを明確にする。
- **(d) 対応不要と説明する**: 誤解なら事実を、対応済みなら該当コミット/箇所を指し示す。

### 投稿前に確認を取る

返信は **外向きの操作** で、関係者に通知され取り消しづらいので、投稿前に必ずユーザーへ全文を提示して了承を取ります。修正指示があれば反映して再提示します。

### 通知を減らす

返信は可能な限り **1 回のレビュー投稿にまとめます**（1 コメントごとに投稿しない）。[gh-review](../gh-review/SKILL.md) と同様、`event` は `COMMENT` を既定とし、勝手に `REQUEST_CHANGES` 等に上げません。特定スレッドへの返信は、まとめて近いタイミングで投稿し、通知の分散を避けます。

## 6. 実行

### push

コードを直した場合、GitHub に反映するため push します。**まず一度 `git push origin <branch>` を試します**（push 先は明示する）。成功すればそのまま進みます。

**push が失敗した場合**（認証・パスフレーズ入力が対話的に必要、権限が無い等）は、無理に繰り返さず、その時点で **ユーザーに push を依頼** します。失敗の内容を簡潔に伝え、実行してほしいコマンド（`git push origin <branch>`）を示します。ユーザー自身のシェルなら対話的な入力もできます。

### 返信の投稿

確認の取れた返信を投稿します。特定のインラインスレッドへ返信するには、そのスレッドの元コメント ID に対して返信します。

```
gh api --method POST /repos/{owner}/{repo}/pulls/<num>/comments/<comment_id>/replies -f body='...'
```

複雑な本文はシェルのエスケープで壊れやすいので、長文・記号を含む返信は `-F body=@<file>` でファイルから渡し、後で片付けます（一時ファイルは作業ツリーを汚さない場所に置く）。

### スレッドの resolve

resolve は GraphQL の `resolveReviewThread` で行います（thread ID は 2 で取得済み）。

```
gh api graphql -f query='mutation { resolveReviewThread(input: {threadId: "<thread_id>"}) { thread { isResolved } } }'
```

**resolve してよいのは、コード修正で応えて push 済みの指摘（a）のスレッドだけです。**

- **押し返した（b）・質問した（c）スレッドは resolve しない。** 相手の返答を待つ。resolve は相手（レビュアー）の権利であり、議論を一方的に打ち切る操作になる。
- **(d) 対応不要と説明したスレッド**も、原則こちらからは resolve せず、説明への相手の反応を待つ（明確な誤解で相手も納得済みと分かる場合を除く）。

## 7. 後処理

- コードを直したら **本文も直す**。[gh-pr-update](../gh-pr-update/SKILL.md) を呼び、PR の title・body・labels を最新の差分に同期する（「コードを直したら説明も直す」の原則）。本スキルはコメントの読解・分類・返信・resolve までを担い、本文の再合成はこのスキルの責務ではなく gh-pr-update に委譲する。
- 一時的に作った返信ファイル等は片付ける。
- 何をしたか（直した指摘・押し返した指摘・待ち状態のスレッド）を簡潔にユーザーへ報告する。

## 判断のガードレール（まとめ）

- **押し返しは 1 往復まで。** 一度根拠を添えて反論し、それでも同じ指摘が再度来たら、議論をエスカレートさせずユーザーに引き取ってもらう。堂々巡りをこのスキルの中で続けない。
- **矛盾する指摘は自分で裁定しない。** ユーザーに提示して判断を仰ぐ。
- **resolve は「直して push した a」限定。** 押し返し・質問・説明のスレッドは相手の返答待ちにする。
- **`[対応必須]` でも盲従はしない** が、押し返すなら必ず根拠 + 問いを添える。

## フォールバック: GitHub MCP

`gh` が使えず GitHub MCP が利用できる環境では、GitHub MCP の対応ツールで同じことを行います。レビューコメント/スレッドの取得、インラインコメントへの返信、スレッドの resolve（`resolve_review_thread`）、レビューの投稿は MCP の対応ツールに読み替えます。その場合も「全部直すのではなく分類する」「分類表と返信の事前確認」「返信は乱発しない」「resolve は直して push した指摘だけ」「push はまず試し失敗したら委譲」「本文更新は gh-pr-update に委譲」という方針は変わりません。
