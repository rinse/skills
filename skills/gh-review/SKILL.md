# GitHub PR 敵対的コードレビュー

GitHub の PR に対して、独立した懐疑的レビュアー（skeptic）として**敵対的検証**を行い、コードの特定箇所へ **inline comment** を付けるためのスキルです。目的は改善提案の列挙ではなく「反証」— PR の主張・前提・実装が成立しない可能性のある箇所を壊しに行きます。

**このスキルは GitHub の PR 専用です。** GitHub の PR でない場合（GitHub リポジトリでない、GitLab 等の別ホスト、ローカル変更のみ等）は、このスキルの対象外なので何もしません。

## 全体の流れ

1. 対象 PR を特定する
2. 差分を取得し、反証の姿勢でレビューする
3. 指摘を inline comment として組み立てる（深刻度・確度タグ + 筋の良い修正）
4. ユーザーに内容を見せて確認を取る
5. レビューを投稿する

## 1. 対象 PR の特定

まず前提として、**`gh` コマンドが無い場合**（`command -v gh` で確認）→ このスキルは `gh` に依存する。未インストールならその旨を伝え、**インストールするかどうかユーザーに尋ねる**。希望する場合は実行環境（OS / パッケージマネージャ）に合った方法でインストールして続行する（方法は環境に応じて判断。公式は https://cli.github.com ）。希望しない場合はここで終了する。「`gh` が無い」ことと「GitHub リポジトリでない」ことは別物なので混同しない。

レビュー対象の PR を確定させます。

- **PR 番号や URL が指定されている場合** → それを使う。
- **指定がない場合** → 現在のブランチに紐づく PR を `gh pr view` で探す。

```
gh pr view --json number,title,headRefName,url
```

結果によって分岐します。

- **PR が見つかった場合** → その PR を対象にする。番号を控えておく（以降 `<PR番号>`）。
- **GitHub リポジトリだが現在のブランチに PR がない場合** → 黙って止まらず、どの PR をレビューするかユーザーに尋ねる（番号 / URL）。「PR が見つからない」ことと「GitHub でない」ことは別物。
- **そもそも GitHub リポジトリでない場合**（`gh` がリポジトリを認識しない／別ホストのリモート等） → このスキルの対象外。その旨を伝えて終了する。

## 2. 差分の取得とレビュー

PR の差分と説明文を取得します。

```
gh pr view <PR番号> --json title,body
gh pr diff <PR番号>
```

### レビューの姿勢: 反証

あなたはこの変更を書いた本人ではなく、独立した懐疑的レビュアーです。PR の主張（説明文・コミットメッセージ・コメント・命名が示す意図）と実装を対象に、「成立しない可能性のある箇所」を探しに行きます。

1. **作者の意図を好意的に補完しない。** 判断材料は diff・リポジトリの実コード・一次情報のみ。PR 説明文の主張はそれ自体が反証対象（「説明はこう言っているが diff はそうなっていない」を探す）。
2. **「良い点」は書かない。** 成立しない可能性のある箇所だけを挙げる。褒め言葉は反証の密度を薄める。
3. **事実に関する主張は裏を取る。** 外部 API・ライブラリの仕様、数値、性能などの主張は、可能な限り一次情報（公式ドキュメント、Web 検索、リポジトリ内の実コード、手元での実行結果）に当たって確認する。**diff だけで判断しない** — 呼び出し元・呼び出し先・型定義など周辺コードを必要なだけ読む（`gh api` でのファイル取得や checkout で）。確認できなければ「未検証」と明記し、推測で断定しない。
4. **怪しい臭いは見逃さない。** 確信が持てない指摘も捨てずに出す。偽陽性への配慮は「出さない」ことではなく「確度: 低 と正直にラベルする」ことで行う — 見逃された欠陥は本番で発覚するが、確度ラベル付きの低確度指摘は著者が数秒で棄却できる。非対称なので、迷ったら出す側に倒す。

典型的な「怪しい臭い」の観点（網羅ではなく想起の補助。コードの性質に応じて広げる）:

- 境界条件（空・0・null・上限・オフバイワン）とエラーパスの握りつぶし
- 説明文と diff の食い違い、diff に紛れ込んだ無関係な変更
- 変更された挙動に対するテストの不在、既存テストの弱体化
- 並行性・リソースリーク・部分的失敗時の不整合
- 後方互換性を黙って壊す変更（公開 API・スキーマ・設定・シリアライズ形式）
- セキュリティ（入力検証の欠落、秘密情報の混入、権限の拡大）

**重要 — コメントを付けられる行:** inline comment は **diff のハンク内に現れる行**にしか付けられません（変更行および前後のコンテキスト行）。差分に含まれない行を指定すると API が 422 で失敗します。`gh pr diff` の出力を見て、コメント対象の行が差分内にあることを確認してください。反証の材料がハンク外のコード（周辺コードとの矛盾など）にある場合は、ハンク内の関係する行にコメントを付け、本文でハンク外の根拠を引用します。

行番号の数え方:
- ハンクヘッダ `@@ -a,b +c,d @@` の意味: 旧ファイル側は行 `a` から `b` 行分、新ファイル側は行 `c` から `d` 行分が対象。
- **新ファイル側（RIGHT）の行番号** → `c` から数え始め、コンテキスト行（行頭が空白）と追加行（行頭が `+`）だけをカウントする（削除行 `-` は数えない）。
- **旧ファイル側（LEFT）の行番号** → `a` から数え始め、コンテキスト行と削除行（行頭が `-`）だけをカウントする（追加行は数えない）。
- **追加行・コンテキスト行**（新しい側）→ 新ファイル側の行番号を使い、`side` は `RIGHT`。
- **削除行**（古い側）→ 旧ファイル側の行番号を使い、`side` は `LEFT`。
- **複数行にまたがる指摘** → 開始行を `start_line`、終了行を `line` で指定する。
- 投稿前に、各コメントの対象行が上記の数え方でいずれかのハンクの範囲内に収まっているかを検算する。

差分が非常に大きい場合、差分の読解（ファイルごとの変更概要の把握、疑わしい箇所の列挙といった事実の抽出）だけを Agent ツールで安価な Subagent（`model: "haiku"` 等）に委譲してよい。ただし**反証の採否・深刻度・確度の判断は必ずレビュアー自身が行う**（gh-release スキルと同様、判断は手元に残す）。

### 再レビュー時: 既存コメントの確認

同じ PR を二度目以降にレビューする場合は、レビューを組み立てる前に既存のインラインコメントを確認し、重複投稿を避けます。

```
gh api /repos/{owner}/{repo}/pulls/<PR番号>/comments --paginate
```

- 既に指摘済みの点は繰り返さない。
- 前回の指摘がまだ解決されていない（コードが直っていない）場合も、それは「新規の指摘」ではなく「未対応のまま残っている既存の指摘」として扱い、同じ内容を新規指摘として重複投稿しない。

## 3. 指摘のフォーマット

各 inline comment の本文 (`body`) は、必ず次の形式で書きます。

```
[深刻度:高|中|低][確度:高|中|低] <何が成立しない可能性があるか（一言）>

- 根拠: なぜそう言えるか。一次情報を確認した場合はその出典。裏を取れなかった場合は「未検証」と明記する。
- 筋の良い修正: 代わりにどう直すのが筋が良いか。方針レベルでよいが、短く書けるならコード例を添える。
```

**深刻度** — 指摘が正しかった場合の影響:
- `高` — 結論が覆る（この変更がそのままではマージに値しない: バグ、セキュリティ問題、説明と実装の重大な食い違い）
- `中` — 修正が必要（変更の骨子は成立するが、この点は直さないと問題が残る）
- `低` — 条件つきで成立（特定の前提が崩れたときだけ問題になる。前提を明記する）

**確度** — 指摘自体がどれだけ確かか:
- `高` — 一次情報または実コードで裏を取った
- `中` — 状況証拠は揃っているが直接の確認はしていない
- `低` — 推測を含む（それでも出す。著者が数秒で棄却できるように根拠を具体的に書く）

深刻度と確度は独立の軸です。「確度: 低 だから深刻度も下げる」といった混同をしないこと。「深刻度: 高 / 確度: 低」（もし当たっていれば致命的だが未確認）はまさに著者に見てほしい指摘です。

**例:**

- `[深刻度:高][確度:高] リトライ時に冪等性が壊れる可能性` + 根拠（実コードで POST が二重送信になるパスを確認）+ 筋の良い修正（冪等キーの付与）
- `[深刻度:中][確度:低] このライブラリ関数は空配列で例外を投げる可能性（未検証）` + 根拠（ドキュメント未確認、シグネチャからの推測）+ 筋の良い修正（早期 return でガード）

## 4. 投稿内容の確認

レビューの投稿は**外向きの操作**で、PR の関係者に通知され取り消しづらいので、投稿前に必ずユーザーに内容を見せて了承を取ります。

提示する内容:
- 全体サマリー（後述の「壊せなかった点」を含む）
- 各 inline comment の「ファイル:行」「深刻度・確度」「本文」の一覧

ユーザーが了承したら次へ進みます。修正指示があれば反映してから再提示します。

## 5. レビューの投稿

inline comment は1件ずつ投稿するのではなく、**1つのレビューにまとめて**投稿します（通知が1回で済み、レビューとして筋が通るため）。`POST /repos/{owner}/{repo}/pulls/<PR番号>/reviews` を使います。`{owner}` と `{repo}` は `gh` がカレントリポジトリから自動補完します。

レビュー本文 (`body`) の末尾には、**「反証を試みたが壊せなかった点」を1〜2行**で書きます（なければ「なし」）。これは褒め言葉ではなく検証範囲の報告です — どこを攻めて壊せなかったかが分かると、著者は残りの検証を自分でやる必要がなくなります。

コメントの多い JSON はコマンドラインで組むと壊れやすいので、JSON ファイルを作って `--input` で渡します。

`review.json` の例:

```json
{
  "event": "COMMENT",
  "body": "反証の観点でレビューしました。指摘は各行のコメントを参照してください。\n\n反証を試みたが壊せなかった点: エラーパスの網羅とトランザクション境界は攻めたが破綻を見つけられなかった。",
  "comments": [
    {
      "path": "src/auth.py",
      "line": 42,
      "side": "RIGHT",
      "body": "[深刻度:高][確度:高] 空入力で例外になり、認証全体が落ちる可能性\n\n- 根拠: この分岐に null チェックがなく、呼び出し元 `login_handler` (src/handlers.py:88) は未検証の入力をそのまま渡している（実コードで確認）。\n- 筋の良い修正: この関数の入口で空・null を早期 return で弾く。呼び出し元に検証を足すより、この関数が自分の前提を自分で守る方が筋が良い。"
    },
    {
      "path": "src/util.py",
      "start_line": 10,
      "line": 15,
      "side": "RIGHT",
      "body": "[深刻度:低][確度:低] 大きな入力で O(n^2) になる可能性（未検証）\n\n- 根拠: ループ内で線形探索している。入力サイズの上限は確認できていないため未検証。\n- 筋の良い修正: 探索対象を事前に set 化する。入力に上限があるならその根拠をコメントで残す。"
    }
  ]
}
```

投稿コマンド:

```
gh api --method POST /repos/{owner}/{repo}/pulls/<PR番号>/reviews --input review.json
```

### event は COMMENT を既定にする

`event` は `COMMENT` を既定とします。`深刻度: 高` の指摘があっても、勝手に `REQUEST_CHANGES` に上げないこと。**PR をブロックするかどうかはユーザーの判断**であり、スキルが先回りして決めるべきではありません。明示的に「変更を要求して」「approve して」と言われた場合のみ `REQUEST_CHANGES` / `APPROVE` を使います。

### 投稿後

レビューの URL（`gh api` の応答に含まれる `html_url`）をユーザーに伝えます。一時的に作った `review.json` は片付けます。

## フォールバック: GitHub MCP

`gh` が使えず GitHub MCP が利用できる環境では、GitHub MCP の PR レビュー作成ツール（保留レビューの作成 → inline コメントの追加 → レビューの送信、もしくはレビューの一括作成）で同じことを行います。その場合も「line/side に基づく inline コメント」「深刻度・確度タグ + 筋の良い修正」「投稿前の確認」「event は COMMENT 既定」という方針は変わりません。
