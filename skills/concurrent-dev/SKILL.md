# Concurrent Dev

複数の開発タスクを **branch 作成 → worktree 作成 → 開発（Subagent）→ レビュー → マージ → worktree 削除** のパイプラインで並列に進めるスキルです。オーケストレーター（Fable 5 / Opus 級の高等モデル）での利用を想定しています。

## このスキルの核

**開発作業だけを想定難易度に応じて Sonnet / Opus の Subagent に委譲し、コストを削減する**こと。それ以外 ── 準備（ブランチ・worktree の作成）、レビュー、マージ、後片付け ── はすべてオーケストレーターが自分で行います。修正全体を一人が把握し続けることが、並列開発で崩れがちな一貫性（命名・設計・重複の排除）を守る要です。

- **オーケストレーター（あなた）**: 計画、worktree 準備、Subagent への指示、レビュー、マージ、後片付け。
- **Subagent（Sonnet / Opus）**: 割り当てられた worktree 内での実装・テスト・コミットのみ。

## worktree は `.wt/` 下に作る

worktree は必ず**プロジェクトルート直下の `.wt/`** に作ります。Claude Code の許可（ファイルアクセス・コマンド実行）はプロジェクトディレクトリに対して与えられているため、その配下に worktree を置くことで Subagent が許可を適切に引き継げます。プロジェクト外（`/tmp` や Agent ツールの `isolation: "worktree"` が作る場所）に作ってはいけません。

## 全体の流れ

1. 前提確認
2. タスク分解と実行計画（依存関係・難易度・モデル割り当て）
3. 準備: ブランチと worktree の作成
4. 開発: Subagent への委譲
5. レビュー: オーケストレーターによる差分精査
6. マージと検証
7. 後片付け: worktree・ブランチの削除

## 1. 前提確認

- git リポジトリであること。
- 作業ツリーが綺麗であること。未コミット変更があれば、先に退避（コミット / stash）するようユーザーに確認する。
- **ベースブランチ** = スキル起動時にチェックアウトされているブランチ。全タスクはここから分岐し、ここへマージで戻る。
- test / build / lint コマンドを特定しておく（レビューとマージ後検証で使う）。

## 2. タスク分解と実行計画

依頼をタスク単位に分解し、**着手順序と並列度を自分で判断**します。決めることは 4 つ:

- **依存関係**: タスク B がタスク A の成果（API・型・ファイル）を前提とするなら、B は A のマージ後に着手する（= 直列）。依存のないタスク同士だけを並列にする。**全タスクが一本の鎖なら、無理に並列化せず素直に直列で進める**（このスキルのパイプラインは直列でもそのまま使える）。
- **競合の予測**: 同じファイルを大きく触るタスク同士は、依存がなくてもマージ時に衝突する。重なりが大きければ直列に倒すか、タスクの切り方を変える。
- **優先順位**: 依存の根（他をブロックしているもの）と高優先度のものから着手する。
- **モデル割り当て**: タスクごとに想定難易度を見積もり、Sonnet / Opus を割り当てる。

**モデル割り当ての目安:**

| モデル | 向いているタスク |
|---|---|
| Sonnet | 仕様が明確で局所的な変更、既存パターンに倣う実装、機械的なリファクタ、定型的なテスト追加 |
| Opus | 設計判断を含む変更、複数モジュールにまたがる変更、仕様に曖昧さが残るもの、厄介なデバッグ、並行処理・性能まわり |

迷ったら Opus に倒すより先に、**Sonnet + 具体的な指示**を検討する。Sonnet に出すプロンプトは、変更対象ファイル・倣うべき既存コード・受け入れ条件まで具体的に書き下す（後述のテンプレート参照）。それでも判断の余地が大きく残るなら Opus。

計画はこういう表でユーザーに提示し、確認を取ってから着手します:

| # | タスク | ブランチ | モデル | 依存 | 波 |
|---|---|---|---|---|---|
| 1 | retry ユーティリティの追加 | `add-retry-util` | Sonnet | — | 1 |
| 2 | エクスポート機能の設計と実装 | `csv-export` | Opus | — | 1 |
| 3 | fetcher を retry 対応に | `fetcher-retry` | Sonnet | #1 | 2 |

「波」= 同時に走らせるグループ。波 N のタスクは、依存する波 N−1 のタスクがマージされた**あとの最新ベース**から分岐させます。

## 3. 準備: ブランチと worktree の作成（オーケストレーター）

最初に一度だけ、`.wt/` が追跡されないようにします。リポジトリを汚さないよう `.gitignore` ではなく `.git/info/exclude` に入れるのが基本です（既に ignore 済みなら不要）:

```
git check-ignore -q .wt || echo '.wt/' >> .git/info/exclude
```

波ごとに、タスク分のブランチと worktree をまとめて作ります（ブランチ作成と worktree 作成は `-b` で一度に行えます）:

```
git worktree add .wt/<slug> -b <slug> <ベースブランチ>
```

`<slug>` はタスク内容を表す短い kebab-case（例: `add-retry-util`）。worktree のディレクトリ名とブランチ名を一致させておくと管理が楽です。

## 4. 開発: Subagent への委譲

同じ波のタスクは Agent ツールで**同一ブロックから並列に**起動します。`subagent_type: "general-purpose"`、`model` に計画どおり `"sonnet"` / `"opus"` を指定します。波が長くなりそうな場合は `run_in_background: true` で起動し、完了したものから先にレビューに入って構いません。

プロンプトには最低限これを含めます:

- **作業場所**: worktree の絶対パス（例: `/path/to/project/.wt/add-retry-util`）。**この中だけで作業**し、外（メインの checkout や他の worktree）のファイルには一切触れないこと。git 操作もこの worktree 内でのみ行うこと。
- **タスク内容**: 何を作るか、受け入れ条件、（Sonnet なら特に）変更対象ファイル・倣うべき既存コードの場所。
- **検証**: 実行すべき test / build / lint コマンドと、緑にしてから完了すること。
- **コミット**: 完了時に worktree 内でコミットすること。[git-commit](../git-commit/SKILL.md) の規約に従い、一行メッセージ・ファイル明示列挙（`git add -A` 禁止）。push はしない。
- **報告**: 変更の要約、設計上の判断、残した懸念を最後に報告すること。

> **注:** 並列タスクのテストが共有リソース（固定ポート、ローカル DB、グローバルキャッシュ等)を奪い合う場合、テスト実行だけは衝突します。その兆候があればテストはオーケストレーターがマージ検証時に直列で流す方針に切り替えてください。

## 5. レビュー: オーケストレーターによる差分精査

レビューは**原則オーケストレーター自身**が行います。全タスクの差分を一人が読むことが、並列に作られた変更同士の一貫性（重複実装・命名のずれ・方針の食い違い）を見つける唯一の方法だからです。観点が独立した大量の差分など、必要な場合に限り レビュー Subagent を補助に使って構いませんが、最終判断はオーケストレーターが下します。

ブランチごとに差分全体を読み、worktree 内でテストを流します:

```
git -P diff <ベースブランチ>...<slug>
git -C .wt/<slug> <test / build / lint>
```

見るべきこと: 受け入れ条件を満たしているか、既存コードの慣習に沿っているか、**他の並列タスクと矛盾・重複していないか**（同じユーティリティを別々に実装している、等）。

問題があった場合の修正は規模で使い分けます:

- **小さな修正** → オーケストレーターが worktree 内で直接直してコミットする。
- **やり直しに近い修正** → SendMessage で元の Subagent に差し戻す（コンテキストが残っているため再指示が安い）。Sonnet が要求水準に届かなかった場合は、**Opus に切り替えて再投入**する。

## 6. マージと検証

レビューを通ったブランチから、**一つずつ順に**ベースブランチへマージします。中間コミットを履歴に残さないため、squash merge + 一行メッセージが基本です:

```
git merge --squash <slug>
git commit -m "<一行メッセージ>"
```

- マージ順は依存関係と優先順位に従う。並列タスク同士でも後のマージで衝突しうるが、**コンフリクト解消はオーケストレーターが行う**（全タスクの差分を読んでいるので意図を踏まえて解消できる）。
- **各マージの直後にベースブランチでテストを流す。** テキスト上は衝突しなくても意味的に衝突する（片方が消した API をもう片方が使う等）のは並列開発の典型的な事故で、ここで捕まえる。
- 次の波のタスクは、依存先がマージされた**この最新のベース**から分岐させる（→ step 3 に戻る）。

## 7. 後片付け

マージが済んだタスクから worktree とブランチを片付けます:

```
git diff HEAD <slug>            # 空であること（= 取り込み漏れがない）を確認
git worktree remove .wt/<slug>
git branch -D <slug>            # squash merge では -d が通らないため、上の確認を経て -D
```

全タスク完了後、`git worktree list` で残骸がないことを確認し、`.wt/` が空なら削除します。最後に、各タスクの結果（マージされたコミット、レビューでの修正点、残した懸念）をまとめてユーザーに報告します。

## やらないこと

- **push・PR 作成はしない。** ローカルのベースブランチに揃えるところまで。続けるなら [gh-pr-create](../gh-pr-create/SKILL.md) へ渡す。
- **無理に並列化しない。** 依存で一本鎖になるなら直列で進める。並列化はコスト削減の手段であって目的ではない。
- **開発の中身を Subagent に丸投げしたまま信用しない。** レビューなしのマージはしない。
