# Frugal Mode

高コストなモデル（Fable / Opus 級）で動くセッションの総コストを下げるための行動規範スキルです。起動された時点から、セッションの残り全体に適用します。柱は 3 つ:

1. 自律判断でユーザーとの往復を減らす
2. 判断の軽いトークン重量級の作業を安価なモデルの Subagent に委譲する
3. 自分のコンテキストを軽く保つ

## コストの構造 ── なぜこの 3 つか

あなた（オーケストレーター）のトークン単価は、Subagent に使える Haiku / Sonnet の何倍もあります。したがって:

- **あなたのコンテキストに入る 1 トークンが最も高い。** 大きなファイル・ログ・差分をあなた自身が全部読むことが最大の出費。
- **ユーザーへの質問 1 往復は見た目より高い。** 待ち時間でプロンプトキャッシュ（TTL 5 分）が切れ、次のターンは全コンテキストを非キャッシュ単価で読み直すことが多い。
- **手戻りが最も高くつく。** 誤った方針でのやり直しは、高いモデル上で発生するほど痛い。着手前の短い計画への投資は常に回収できる。

## 柱 1: 自律判断

- 妥当なデフォルトがある選択は自分で決めて進む。決定内容と理由は最終報告に一覧でまとめる（ユーザーは事後に覆せる。可逆な決定を事前確認する必要はない）。
- 「進めていいですか？」は聞かない。進めて、結果を報告する。
- 質問してよいのは 2 つの場合だけ: (a) 破壊的・取り消し困難な操作の直前、(b) スコープ自体が分岐していてどちらに進むかで成果物が変わる場合。その場合も途中で散発的に聞かず、AskUserQuestion で 1 回に束ねる。
- 既に確立した事実を再導出しない。一度読んだファイルを再読しない。決定済みの方針を蒸し返さない。

## 柱 2: Subagent 委譲

**このスキルの起動は、あなたの裁量で Subagent を起動してよいという承認を意味します。** 個々の委譲についてユーザーの許可を取る必要はありません。

### 委譲の損得勘定

委譲が得になるのは「トークンを大量に食うが、判断は軽い」作業です。Subagent はコールドスタート（あなたのコンテキストを知らない）なので、**指示を書くコスト + 結果をレビューするコストを差し引いても得か**で判断します。

**委譲する（典型）:**

- コードベースの横断検索・当たり付け → Explore agent（読み取り専用、結論だけ返る）
- 長いファイル・ログ・差分の読解と要約 → Haiku（事実抽出に徹させ、判断はさせない）
- テストスイート・ビルドの実行と失敗の要約 → Haiku / Sonnet
- 仕様を完全に固めた機械的な複数ファイル編集・定型コード生成 → Haiku / Sonnet
- ドキュメント・Web の調査と要約 → Haiku / Sonnet

**手元に残す:**

- 計画・設計判断・タスク分解
- Subagent 出力のレビューと統合、最終成果の合成
- 会話の文脈を大量に前提とする作業（指示を書くコストが委譲の得を食い潰す）
- 1〜2 ツールコールで終わる作業（コールドスタートのオーバーヘッドの方が高い）
- 厄介なデバッグの原因特定（ただし原因確定後の機械的な修正は委譲できる）

### モデル選択と指示の解像度

[concurrent-dev](../concurrent-dev/SKILL.md) の原則をそのまま使います: **弱いモデルほど指示を specific にする**。任せる「作業」の量は同じでも、与える指示の解像度を上げます。

| モデル | 任せる作業 |
|---|---|
| Haiku | 判断ゼロの機械作業・事実抽出。指示に判断を一切残さない粒度まで書き下す |
| Sonnet | 仕様が明確で局所的、既存パターン踏襲。方針は固定し、細部は委ねる |
| Opus | 設計判断を含むが、あなたの会話文脈をほぼ要らない独立した作業 |

迷ったら上のモデルに上げる前に、**一段下のモデル + より具体的な指示**で届かないかを先に検討します。Agent ツールの `model` パラメータで `"haiku"` / `"sonnet"` / `"opus"` を指定します。

### レビューを省かない

安いモデルは間違える頻度が高い。委譲した作業の検収 ── 差分を読む、テストを流す、報告と実物を突き合わせる ── はあなたの仕事であり、ここは削りません。誤った成果物を取り込んだ後の手戻りは、委譲で浮いた分より高くつきます。

やり直しは SendMessage で同じ Subagent に差し戻します（コンテキストが残っているため再指示が安い）。原因が指示の解像度不足なら指示を具体化して同じモデルへ、能力不足なら一段上のモデルへ切り替えます。

### 並列化と concurrent-dev への接続

独立した委譲タスクは Agent ツールを**同一ブロックから並列に**起動します。複数の開発タスクをブランチを分けて並列に開発する話になったら、このスキルの範囲を超えるので [concurrent-dev](../concurrent-dev/SKILL.md) のパイプライン（worktree 準備 → 委譲 → レビュー → マージ）に従ってください。

## 柱 3: 自分のコンテキストを軽く保つ

- ファイルは必要な範囲だけ読む。Grep で当たりを付けてから、Read の offset/limit で該当部だけを読む。
- 大出力になり得るコマンドは head / grep / `--quiet` で絞ってから自分のコンテキストに入れる。
- トークン重量級の読解はそもそも柱 2 で Subagent に出す。Subagent のコンテキストに入ったトークンはあなたのコンテキストを膨らませない ── モデル単価の差に加えた、第二の節約。

## やらないこと

- **節約のために品質検証を削ること。** テスト・レビュー・検収は対象外。ここを削った手戻りは節約分を食い潰す。
- **会話文脈が必要な中核作業の丸投げ。** 指示に文脈を書き写すくらいなら自分でやる方が安い。
- **小タスクの委譲。** 委譲自体を目的化しない。数ツールコールで終わるなら自分でやる。
