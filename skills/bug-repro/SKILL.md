# Bug Repro (バグの再現)

雑なバグ報告（「〜するとおかしい」「たまに落ちる」）を、**報告された症状を実際に再現する、最小で決定的な失敗テスト**に変換するスキルです。[gh-issue-create](../gh-issue-create/SKILL.md) が「雑な記述 → 構造化された Issue」であるのと対で、これは「雑な報告 → 実行可能な再現」です。

**修正はこのスキルの範囲外です。** 成果物は「壊れていることを示す失敗テスト」まで。修正に進むなら、それを [tdd](../tdd/SKILL.md) の Red の入口として引き渡します。

## このスキルの核: 2 つの規律

このスキルの価値は手順の列挙ではなく、次の 2 つの規律を守り切ることにあります。

1. **最小化** ── 再現できたら、削れるだけ削る。依存・セットアップ・データ・アサーションを最小に絞り、乱数・時刻・並行性を固定して**決定的**にする。「大きくて再現する」テストは道半ばで、削り切って初めて成果物になる。
2. **正直さ** ── 再現できなかったとき、**でっち上げない**。「再現できた風のテスト」を作らず、「再現できなかった」と正直に報告し、試した条件を列挙する。

2 番目が特に肝心です。エージェントは放っておくと **「それらしく失敗するテスト」**（報告されたバグではなく、書き方のミスや別の原因で失敗するだけのテスト）を作り、それを「再現できました」と差し出してしまいがちです。これを禁じる規律こそがこのスキルの存在意義です。失敗の内容が報告された症状と一致していなければ、それは再現ではありません（後述の「3. 再現の判定」）。

## 全体の流れ

1. 報告の理解
2. 仮説と再現の試み
3. 再現の判定（失敗しただけでは再現ではない）
4. 最小化
5. 環境要因の切り分け
6. 成果物と報告

## 1. 報告の理解

報告から、次の 3 点を抽出します。

- **期待動作** ── 本来どうなるはずだったか。
- **実際の動作** ── 何が起きたか（エラーメッセージ、誤った値、クラッシュ、無反応など、観測可能な症状）。
- **発生条件** ── どういう入力・操作・状態で起きるか。「たまに」なら、その「たまに」の手がかり（特定のデータ、タイミング、環境）。

報告に欠けていて、かつ**推測で埋められない**要素（具体的な入力データ、環境・OS、ライブラリやアプリのバージョン、再現の手順）は、先に進む前にユーザーに一つ二つ確認します。ただし尋ねすぎず、コードや報告から妥当に補える部分は自分で埋めます。

## 2. 仮説と再現の試み

関連するコードを読み、発生条件についての仮説を立て、それを突く再現テストを書きます。

- **第一候補は、プロジェクトのテストフレームワークでの失敗テスト**です。既存のテストの置き場所・命名・実行方法（`package.json` の scripts、`pytest.ini`、`Cargo.toml`、`Makefile` など）を確認し、それに倣います。プロジェクトの流儀に乗せておくと、そのまま CI や後続作業（tdd）に渡せます。
- テストフレームワークが無い／使えない場合は、**単体で実行できる再現スクリプト**を書きます。この場合も「6. 成果物と報告」の受け渡し規約（単一コマンド・終了コードで合否）を満たす形にします。

書いたら**実行して観察**します。仮説どおりに失敗しなければ、コードとログを読み直して仮説を更新し、再度試します。この「書く → 走らせる → 観察する → 仮説を直す」を、再現できるか、再現できないと結論づけるまで反復します。実際の失敗出力を見ずに「たぶんこれで再現する」で止めないこと。

## 3. 再現の判定 ── 失敗しただけでは再現ではない

**このスキルで最も重要な判断です。テストが赤くなっただけでは再現とは呼べません。** 失敗の内容が、報告された症状と一致しているかを必ず確認します。

- 出たエラーメッセージ・例外の型・スタックトレース・誤った値が、報告された症状と**同じもの**か。
- テストのセットアップ不備、import ミス、存在しない API の呼び出し、アサーションの書き間違いといった、**バグとは無関係な理由**で失敗していないか。
- 期待動作を誤解して、正しい挙動を「間違い」とアサートしているだけになっていないか。

別の理由で失敗しているなら、それは再現ではありません。その失敗を潰し、報告された症状そのもので失敗する状態に到達して初めて「再現できた」と言えます。ここを曖昧にすると「2 つの規律」の 2 番目（正直さ）が崩れます。

## 4. 最小化

再現を**保ったまま**、削れるものを削ります。各削除のあとに必ず再実行し、まだ同じ症状で失敗することを確認します（削った結果、別の理由で失敗するようになったら削りすぎ）。

- **セットアップ・データ**: 症状の再現に不要な準備・フィクスチャ・入力を取り除く。データは症状を出す最小の形まで縮める。
- **依存**: 症状に関係しない外部依存・モジュールへの依存を外す。
- **アサーション**: **症状を突くもの 1 点に絞る**。ついでの検証を足さない。その 1 つが、まさに報告されたバグを捉えていること。
- **決定性**: 乱数はシード固定、時刻は固定値を注入、並行性は逐次化するなどして、**毎回同じ結果**になるようにする。決定的にできない場合は「5. 環境要因の切り分け」「フレーキーな再現」に従う。

## 5. 環境要因の切り分け

再現が特定の環境条件に依存する場合（ライブラリ・言語のバージョン、タイムゾーン、ロケール、並行性、ネットワーク、OS）は、その条件を**テスト内で固定する**か、固定できないなら**コメントで明記**します。

- 例: タイムゾーン依存なら `TZ` を固定するか、テスト内で明示的に timezone を設定する。ロケール依存なら locale を固定する。バージョン依存ならその version を報告とコメントに残す。
- 目的は、他の人・他の環境・CI で走らせたときに「なぜここでは再現しないのか」で消耗しないよう、再現の前提を成果物自身に埋め込むことです。

## 6. 成果物と報告

成果物は、**失敗するテスト（またはスクリプト）** と、**短い分析**です。分析には、トリガー条件（どういう入力・状態で起きるか）・失敗の仕方（どの症状で落ちるか）・原因についての仮説を書きます。原因の**特定と修正はしません**（tdd に渡す）。

### 下流との受け渡し規約（重要）

再現の成果物は、後続のスキルやツールがそのまま述語として使えるよう、次の契約を満たす形に整えます。

- **単一コマンドで実行でき、終了コード 0 / 非 0 で合否を判定できる**こと。

これは例えば `git bisect run` に述語として渡す、CI で検証する、といった下流の使い方をそのまま可能にするための契約です。テストフレームワーク経由なら、**該当テスト 1 件だけを走らせるコマンド**を必ず報告に含めます。

- Vitest: `npx vitest run -t '<test name>'`
- pytest: `pytest path/to/test_file.py::test_name`
- Cargo: `cargo test <test_name>`
- Go: `go test ./pkg -run '^TestName$'`

再現スクリプトの場合は、その 1 コマンド（例: `node repro.mjs` / `python repro.py`）と、症状が出たとき非 0 で終了することを明記します。

### 再現できなかったときの報告様式

「再現できませんでした」で終わらせないこと。次の一手を渡します。次を列挙します。

- **試した条件** ── 入力・環境（OS / バージョン / タイムゾーン等）・試行回数。何を変えて何を試したか。
- **観察できたこと** ── 実際に起きたこと（エラーが出なかった／別の挙動だった等）。
- **残る仮説** ── まだ試せていない筋、怪しいと見ている箇所。
- **ユーザーに追加で確認したいこと** ── 再現に必要そうな、報告に欠けている情報（正確な入力データ、バージョン、手順、発生頻度）。

これらを揃えることで、ユーザーは不足情報を補って再挑戦させられます。

### フレーキーな再現

毎回は失敗しない（間欠的にしか再現しない）場合は、決定的にできないか（5. 参照）をまず試みます。それでも間欠的なら、**N 回実行して再現率を測り**、報告に含めます（例: 「20 回中 3 回失敗」）。再現率も成果物の一部です。判定（3.）は再現した回の失敗が症状と一致しているかで行います。

### 成果物の置き場所（ユーザーに確認）

再現テストは、修正されるまで**走らせれば失敗し続ける** ── つまりコミットして CI に載せると、CI が赤くなり続けます。そのため、これをどう扱うかは**ユーザーに確認**します。

- **保守的な既定: コミットしない。** 失敗テストを手渡し、修正時に取り込んでもらう。
- 望まれれば、`skip` / `xfail`（Vitest の `test.skip` / `test.fails`、pytest の `@pytest.mark.xfail`、Rust の `#[ignore]` 等）でマークしたうえでコミットする。この場合も、修正時にマークを外して有効化することを申し送る。

いずれにせよ、勝手にコミットして CI を赤くしないこと。`git add` は対象ファイルを明示列挙します（`git add -A` は使わない）。

## まとめ

1. 報告から期待・実際・条件を抽出し、埋められない不足はユーザーに確認する。
2. 仮説を立て、フレームワークの失敗テスト（無ければ再現スクリプト）を書き、走らせて観察し、反復する。
3. **失敗の内容が症状と一致するか**を確認する。別の理由で落ちているなら再現ではない。
4. 再現を保ったまま最小化し、決定的にする。
5. 環境依存は固定するかコメントで明記する。
6. 失敗テスト + 短い分析を、**単一コマンド・終了コードで合否**の形に整えて渡す。再現できなければ**正直に**、試した条件とともに報告する。修正は [tdd](../tdd/SKILL.md) の Red から。
