# GitHub Issue Create

雑な思いつきや要望を、構造化された GitHub Issue に**合成**してから、確認を取って起票するスキルです。タイトルと本文をきれいに組み立てるのがこのスキルの核です。

**このスキルは GitHub の Issue 起票専用です。** GitHub リポジトリでない場合（別ホストのリモート、リモート未設定等）は、このスキルの対象外なので何もしません。

## 全体の流れ

1. ガードと前提確認
2. 入力（やりたいこと）の理解
3. テンプレートの判定
4. タイトルとラベルの生成
5. ユーザーに内容を見せて確認を取る
6. Issue の作成
7. 後処理

## 1. ガードと前提確認

- **`gh` コマンドが無い場合**（`command -v gh` で確認）→ このスキルは `gh` に依存する。未インストールならその旨を伝え、**インストールするかどうかユーザーに尋ねる**。希望する場合は実行環境（OS / パッケージマネージャ）に合った方法でインストールして続行する（公式は https://cli.github.com ）。希望しない場合はここで終了する。「`gh` が無い」ことと「GitHub リポジトリでない」ことは別物なので混同しない。
- **そもそも GitHub リポジトリでない場合**（`gh` がリポジトリを認識しない／別ホストのリモート等）→ このスキルの対象外。その旨を伝えて終了する。

## 2. 入力の理解

ユーザーが渡した記述（バグ報告、機能要望、改善案など）を読み取ります。曖昧で本文を組み立てられない場合は、起票前に不足点を一つ二つ尋ねます。ただし尋ねすぎず、合成できる部分は自分で埋めます。関連するコードがあれば軽く確認し、制約や受け入れ条件を具体的にします。

## 3. テンプレートの判定

リポジトリに Issue テンプレートがあれば、それに沿って本文を書きます。次の場所を確認します（大文字小文字の違いも含む）。

- `.github/ISSUE_TEMPLATE/`（複数テンプレートのディレクトリ。`*.md` / `*.yml`）
- `.github/ISSUE_TEMPLATE.md`
- リポジトリ直下の `ISSUE_TEMPLATE.md`
- `docs/ISSUE_TEMPLATE.md`

- **テンプレートがある場合** → その各セクションを、理解した内容で埋める。空欄やコメント（`<!-- -->`）のプレースホルダはそのまま残さず実際の内容に置き換える。複数テンプレートがある場合は、内容に最も合うものを選ぶ（迷えばユーザーに尋ねる）。YAML フォーム（issue forms）の場合は各フィールドの趣旨に沿って本文を構成する。
- **テンプレートが無い場合** → 次のフォールバックテンプレートを使う。

### フォールバックテンプレート

各セクションの括弧内は**書き方の指針**であり、そのまま本文に残さず実際の内容で置き換えます。

```markdown
## 概要
(数行で短く。形式的・構造的である必要はなく、コンセプトが伝わる文章にする)

## 目的
(箇条書き。各項目は一行であるのが望ましい)

## 制約
(実装上の制約、ユーザーの好み、既に分かっている問題点など、避けるべき点)

## 受け入れ条件
(箇条書き。理想的には、これらをすべて確認できたら即マージできる、という性質のもの)
```

各セクションを埋めるときの判断指針:

- **目的** が複数行に膨らむ場合は、Issue を分割するか Epic Issue にまとめることを検討し、その旨をユーザーに提案する（一つの Issue に詰め込みすぎない）。
- **受け入れ条件** は可能な限り自動テストにできる形で書く。テストの優先順位は **Unit test > 統合テスト > E2E テスト**。これを起点に TDD で進められるとぶれにくい。ただし実際にはコードレビュー・アーキテクチャレビュー・DB スキーマレビュー等の人手の確認が必要になる条件も多いので、それらは自動テスト化を強いず条件として明記する。

## 4. タイトルとラベルの生成

タイトルは、課題の主題を端的に伝える一行にします。

- バグなら現象を、要望なら達成したいことを表す。
- 必要なら種別の prefix を付ける（`bug:` / `feat:` など。リポジトリの慣習があればそれに合わせる）。

### ラベルの選定

Issue の内容に合った**適切なラベルを自分で選び**、付与します。手順:

1. リポジトリに存在するラベルを `gh label list` で取得する（名前だけでなく description も読む。各ラベルの用途は description から判断する）。
2. その中から、Issue の内容に合致するものを選ぶ。観点の例:
   - **種別**: `bug` / `enhancement` / `feature` / `documentation` / `refactor` など、課題の性質に合うもの。
   - **領域・コンポーネント**: `area/*` `component/*` のように対象範囲を表すラベルがあれば、関係するものを付ける。
   - **優先度**: リポジトリが `priority/*` 等で運用していて、内容から優先度が読み取れる場合のみ。読み取れないなら付けない。
3. **既存のラベルだけを使う。** リポジトリに無いラベルは指定しない（`gh issue create` が失敗する）。明らかに必要なラベルが存在しない場合は、勝手に作らず、その旨をユーザーに伝えて作成するか確認する。
4. **付けすぎない。** 内容を正確に表す最小限に絞る。確信を持って当てはまるものだけを選び、迷うものは外す。合致するラベルが無ければ無理に付けない。

選んだラベルは、理由が自明でなければ一言添えて、step 5 でユーザーに提示します。

### AI 作成ラベルの付与

この Issue はこのスキル（AI）によって作成されるため、**AI による作成を示すラベル**があれば付与します。上の「ラベルの選定」と同じく、**既存ラベルを使い、無ければ勝手に作らずユーザーに確認する**保守的な方針です（AI ラベルだけは Issue の内容ではなく「AI が作った」という事実に基づき、常に付与対象になる点が違います）。

1. step 4 で取得した `gh label list` の結果から、**AI による作成を示すラベルが既にあるか**を意味で判断して探す。ただし**保守的に**寄せ、明確に「AI／自動生成で作られたもの」を表すラベルだけを対象にする（例: `AI ✨`、description に "created by AI" / "AI-generated" とあるもの）。`bot` / `automated` のように AI 以外（Dependabot 等）も指しうる曖昧なラベルは対象にしない。判断に迷うものは「無い」とみなして次へ進む。
2. **既にある場合** → そのラベルを付与対象に加える（step 5 の確認でまとめて提示する）。
3. **無い場合** → 下記ラベルを新規作成して付与してよいか**ユーザーに尋ねる**。了承されたら作成して付与対象に加える。断られたら付けずに進む（Issue 作成自体は止めない）。
   - Name: `AI ✨`
   - Description: `This was created by AI, either at a user's request or autonomously.`
   - Color: `333333`（`#` は付けない）
   - 作成: `gh label create "AI ✨" --description "This was created by AI, either at a user's request or autonomously." --color 333333`
4. **断られた場合は記憶を試みる。** 同じリポジトリで毎回尋ねないよう、利用できる Memory 機能があれば「このリポジトリでは AI ラベルを追加しない」ことを**リポジトリ単位で**記録する。**ただしそのために新しいファイルを作ったり、リポジトリ／作業ツリーに変更を加えたりはしない**（あくまでリポジトリ外の記憶機構を使う。記憶手段が無ければ諦めてよい）。そして次回以降は step 3 で尋ねる前にこの記憶を確認し、記録があれば作成・質問を省略する。
5. **ユーザーに質問できない非対話環境（autopilot 等）** では、AI ラベルが既にあれば付与し、**無い場合の作成・質問は省略**する（勝手に新規ラベルを作らない）。

## 5. 起票内容の確認

Issue の作成は**外向きの操作**で、ウォッチャーに通知され取り消しづらいので、作成前に必ずユーザーに内容を見せて了承を取ります。

提示する内容:

- タイトル
- 本文
- **付けるラベル**（step 4 で選定したもの。ユーザーは追加・削除を指示できる）
- 担当者・マイルストーンがあればそれも（既定では付けない。ユーザーが希望した場合のみ）

ユーザーが了承したら次へ進みます。修正指示（ラベルの変更含む）があれば反映してから再提示します。

## 6. Issue の作成

本文はコマンドラインに直書きすると壊れやすいので、ファイルに書き出して `--body-file` で渡します。

```
gh issue create --title "..." --body-file issue-body.md --label "<label1>" --label "<label2>"
```

- step 4 で選び、確認の取れたラベルを `--label` で付ける。存在しないラベルを指定すると失敗するので、選定時に `gh label list` で実在を確認しておく。
- 担当者・マイルストーンはユーザーが希望した場合のみ `--assignee` / `--milestone` で付ける。

## 7. 後処理

作成された Issue の URL をユーザーに伝えます。一時的に作った本文ファイル（`issue-body.md` 等）は片付けます。

## フォールバック: GitHub MCP

`gh` が使えず GitHub MCP が利用できる環境では、GitHub MCP の Issue 作成ツールで同じことを行います。その場合も「記述から本文を合成」「テンプレートがあれば従う」「内容に合うラベルを選定して付与」「作成前の確認」という方針は変わりません。ラベルの一覧取得・付与も MCP のラベル系・Issue 作成ツールで行います。
