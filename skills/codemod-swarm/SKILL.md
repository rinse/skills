# Codemod Swarm

数十〜数百ファイルにまたがる機械的な変更（API 移行、非推奨パターンの一掃、ライブラリ差し替え、正規表現では書けない一括書き換え）を、スクリプト、Haiku Subagent、オーケストレーターの三層で分担して一掃するスキルです。オーケストレーター（Fable 5 / Opus 級の高等モデル）での利用を想定しています。

## このスキルの核

**自動化境界の三分割**がこのスキルの核です。同じ「機械的な変更」に見えても、どこまでを人手・LLM なしで片付けられるかは箇所ごとに違います。

1. **スクリプト層**：置換が構文的に一意に決まる部分です。sed / perl / ast-grep / comby やエコシステムの codemod ツールで処理し、LLM は使いません。
2. **Haiku 層**：変更の型は同じでも、各箇所の文脈に合わせた微調整（変数名、引数の組み替え、import の位置など）が要る部分です。オーケストレーターが自分の手で作った模範変換の diff を few-shot にして、Haiku の Subagent に分担させます。
3. **残渣層**：パターンの例外、判断が要る箇所です。オーケストレーター自身（または Sonnet）が処理します。

どの箇所をどの層に振るかの判定と、Haiku 層に渡す模範・指示の質が、機械的なコマンドでは埋められない判断です。

## 全体の流れ

1. 前提確認
2. インベントリ
3. 三分割の判定と計画の確認
4. 模範変換（オーケストレーター自身）
5. スクリプト層の実行
6. Haiku 層の実行
7. 残渣層の処理
8. レビュー・検証・コミット
9. 報告

## 1. 前提確認

- git リポジトリであること、作業ツリーが綺麗であることを確認します。
- test / build / lint コマンドを特定しておきます。
- ベースラインが green であることを確認します。あとで失敗が出たとき、変更起因の失敗と既存の失敗を切り分けるためです。

## 2. インベントリ

grep / ast-grep 等で対象箇所を全列挙し、件数とディレクトリ別の分布を把握します。

```
ast-grep -p 'oldFn($$$ARGS)' --lang ts | wc -l
grep -rl 'oldFn(' src | xargs -n1 dirname | sort | uniq -c
```

ここで確認すべきことは一つです。**変更ルールを一〜数文で書き下せるか。** 書き下せないなら、それは機械的な変更ではなく設計判断を含む開発です。このスキルの対象外で、concurrent-dev の領分になります。

## 3. 三分割の判定と計画の確認

インベントリを三層（スクリプト層 / Haiku 層 / 残渣層）に振り分け、層別の件数付きの計画をユーザーに提示して確認を取ります。

判断の指針はシンプルです。まずスクリプト層で何割消せるかを検討します。**決定的にできる作業を LLM にやらせません。** 構文的に一意なパターンはスクリプトの領分であり、Haiku 層に回すのは文脈依存の微調整が要るものだけです。

提示する計画は、層ごとの件数と処理手段がわかる表にします。

| 層 | 件数 | 処理手段 |
|---|---|---|
| スクリプト層 | 240 | `comby` で一括置換 |
| Haiku 層 | 85 | 12 シャードに分割し few-shot で処理 |
| 残渣層 | 9 | オーケストレーターが個別に判断 |

## 4. 模範変換（オーケストレーター自身）

代表的なファイルを 2〜3 個、オーケストレーター自身が自分の手で変換し、テストを流して正しさを確認します。代表には「典型例」に加えて「変則例」を 1 つ含めます。

この diff がそのまま Haiku への few-shot になります。模範の質が、そのまま Haiku 層全体の出力の質を決めます。

## 5. スクリプト層の実行

スクリプトを実行し、typecheck / test で検証し、コミットします。

## 6. Haiku 層の実行

この工程が最も分量を要します。

**シャーディング。** ファイル単位で互いに素に分割します。分割境界はファイル数ではなく、モジュールやディレクトリの境界に合わせます。同じモジュール内のファイルを同じシャードにまとめたほうが、Haiku が文脈を一貫して保てるためです。1 シャード 5〜15 ファイルが目安です。

**並列実行はメインの checkout 内で行い、worktree を作りません。** シャードが互いに素なので編集は衝突しないためです。独立タスクを worktree で隔離する concurrent-dev とは、ここが構造上の違いです。concurrent-dev の各タスクは互いに何を変更するか予測できないのに対し、codemod-swarm のシャードは最初からファイル単位で不可侵な範囲を切ってあるので、worktree による隔離が要りません。

**Subagent には git 操作とテスト実行をさせません。** 並列に走らせると git の index やビルドの成果物が衝突するためです。検証とコミットは、波（並列に走らせる Haiku Subagent の一群）ごとにオーケストレーターが行います。

Haiku への指示には、次をテンプレートとして必ず含めます。

- 模範変換の diff をそのまま貼る。
- 担当するファイルを明示的に列挙する。
- 「パターンに合わないファイルは触らず、スキップとして報告する」という出口を必ず与える。
- git 操作禁止、テスト実行禁止を明記する。
- 最後に、変更したファイルとスキップしたファイルを報告させる。

このうち**「合わないものは触らない」が最重要のルールです。** 弱いモデルに例外の処理までやらせると、無理に模範を当てはめて壊します。例外はすべて残渣層に回収し、判断できる者だけが処理する体制を崩さないことが、Haiku 層を安全に並列化できる理由です。

## 7. 残渣層の処理

Haiku から報告されたスキップ分と、最初から残渣層に振った分を、オーケストレーターが処理します。

量が多く、かつ修正の型が定まっている場合は Sonnet に切り出して構いませんが、どの型を当てるかの判断はオーケストレーターに残します。

## 8. レビュー・検証・コミット

機械的な差分のレビューは、全ファイルを同じ密度で読みません。各シャードから数ファイルを精読し、残りは diff stat の外れ値（異常に大きい、または異常に小さい差分）を探します。パターンに正しく適合しているかを確かめる読み方であり、内容の一語一句を追う読み方ではありません。

波ごとに typecheck / test を流し、緑を確認してからコミットします。コミットは git-commit スキルの規約（一行メッセージ、ファイルの明示列挙、`git add -A` 禁止）に従います。

**層・波ごとにコミットを分けます。** bisect で問題の混入箇所を層・波単位まで絞り込めるようにしておくためです。

## 9. 報告

作業の最後に、次を報告します。

- 層別の処理件数（スクリプト層 / Haiku 層 / 残渣層）。
- スキップ・例外の一覧と、それぞれの処理結果。
- 検証結果（typecheck / test の結果）。

## やらないこと

- **push・PR 作成はしない。** ローカルに層・波ごとのコミットを積むところまでです。続けるなら gh-pr-create へ渡します。
- **変更ルールが書き下せないタスクには使わない。** それは設計判断を含む開発であり、concurrent-dev の領分です。
- **決定的にできる作業を LLM にやらせない。** 構文的に一意な置換はスクリプト層で片付け、Haiku や Sonnet には文脈依存の微調整と判断だけを回します。
