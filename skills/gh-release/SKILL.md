# GitHub Release

最後のリリース以降の差分を読み、**次のセマンティックバージョンを判断**し、リリースノートを**合成**して、注釈付きタグを作成（任意で GitHub Release も作成）するスキルです。バージョンの決定とノートの生成という、機械的なコマンドでは埋められない判断がこのスキルの核です。

トークンを食いやすい**差分の読解だけは Haiku の Subagent に委譲**し、自分（オーケストレーター）はコミットログを読み、Subagent の報告と突き合わせて bump 種別を判断します。判断・ノート合成・タグ作成・確認はすべてオーケストレーターが行い、Subagent には差分からの事実抽出だけを任せます。

バージョン体系は [Semantic Versioning 2.0.0](https://semver.org/) に従います。タグ形式は既定で `v` を付けない `X.Y.Z`（最初のリリースは `0.1.0`）。ただしリポジトリに既存タグがあれば、その規約（prefix の有無）に合わせます。

## 全体の流れ

1. ガードと前提確認
2. 前バージョン（直近のタグ）の特定
3. コンテキストの収集（ログは自分、差分は Haiku の Subagent）
4. バンプ種別の判断と次バージョンの算出（ログ + Subagent の報告）
5. リリースノートの生成
6. ユーザーに内容を見せて確認を取る
7. 注釈付きタグの作成
8. タグの push（ユーザーに依頼）
9. GitHub Release の作成（任意）
10. 後処理

## 1. ガードと前提確認

- **git リポジトリでない場合** → このスキルの対象外。その旨を伝えて終了する。
- **タグ付け対象がデフォルトブランチでない場合** → 通常リリースはデフォルトブランチ（多くは `main`）から切る。別ブランチに居る場合は、その旨を伝えて本当にそこから切るかユーザーに確認する。
- **作業ツリーに未コミットの変更がある場合** → タグはコミットを指すので、未コミットの変更はリリースに含まれない。その旨を伝え、先にコミットするか今の HEAD で進めるかをユーザーに確認する。
- **`gh` コマンドの有無**（`command -v gh`）→ タグ作成だけなら `gh` は不要。GitHub Release も作る場合のみ必要。`gh` が無く Release も希望する場合は、その旨を伝えインストールするか尋ねる（希望すれば環境に合った方法で。公式は https://cli.github.com ）。タグ作成だけで十分なら `gh` 無しでも続行する。

## 2. 前バージョンの特定

直近のリリースタグを特定します。

```
git tag --list --sort=-version:refname
git describe --tags --abbrev=0 2>/dev/null
```

- **タグがある場合** → セマンティックバージョンとして最大のものを前バージョンとする。既存タグの prefix 規約（`v` の有無）を読み取り、新タグもそれに合わせる。
- **タグが一つも無い場合** → 初回リリース。既定の初期バージョンは `0.1.0`。ユーザーが別の初期バージョンを指定すればそれに従う。

以降、前バージョンを `<prev>`、これから打つバージョンを `<next>` とします。

## 3. コンテキストの収集

次バージョンとノートを「実際の変更内容」から組み立てるため、前バージョン以降の **ログは自分（オーケストレーター）で読み、差分は Haiku の Subagent に読ませて報告を受け取ります**。差分は大きくなりがちでトークンを食うため、読解だけ安価な Haiku に委譲し、**最終的な bump 判断はログと Haiku の報告を突き合わせて自分が下します**。

まずログを自分で読みます。

```
git --no-pager log <prev>..HEAD
```

**前バージョン以降にコミットが無い場合**（`git log <prev>..HEAD` が空 = HEAD が既に最新タグ）→ 切るべき新バージョンが無い。その旨を伝えて終了する（現行バージョンは `<prev>`）。Subagent は起動しません。

コミットがある場合、差分の読解を Agent ツールで Haiku の Subagent に委譲します（`subagent_type: "general-purpose"`、`model: "haiku"`）。Subagent には次を指示します。

- 実行するコマンド: `git -P diff <prev>..HEAD`（初回でタグが無い場合は全体の差分。範囲は Subagent のプロンプトに具体的に書き下す）。
- 報告してほしいこと（**判断はこちらで行うので、Subagent には事実の抽出だけを求める**）:
  - 変更を「機能追加 / 挙動変更 / バグ修正 / 内部リファクタ / ドキュメント・テストのみ」に分類した要約（箇条書き、利用者目線で）。
  - **後方互換を壊しうる変更**（公開 API の削除・シグネチャ変更、設定・データ形式の非互換変更、サブコマンドの削除等）を、根拠となるファイル・関数名つきで列挙。無ければ「無し」と明記。
  - リリースノートの素材として使える、ユーザー影響のある変更点の一覧。
- Subagent には bump 種別やバージョンを**決めさせない**（事実報告に徹させる）。

Subagent の報告を受け取ったら、それと自分が読んだログを突き合わせます。報告だけで破壊的変更の有無に確信が持てない場合は、該当ファイルの差分を自分で `git -P diff <prev>..HEAD -- <path>` でピンポイントに確認してから判断します。

## 4. バンプ種別の判断と次バージョンの算出

自分が読んだログと Subagent の差分報告を併せ、変更全体のうち**最も大きい変化**を見極めて bump 種別を決めます。コミットの prefix は手がかりですが、最終判断は差分の実体（Subagent が挙げた破壊的変更の有無等）に基づきます（雑な prefix に引きずられない）。

| 種別 | 目安 |
|------|------|
| **major** | 後方互換を壊す変更（公開 API の削除・シグネチャ変更、設定・データ形式の非互換変更、削除されたサブコマンド等）。コミットの `!` 付き prefix（`feat!` 等）や `BREAKING CHANGE` も手がかり。 |
| **minor** | 後方互換を保つ機能追加（`feat` / `feature`、新しいオプションやエンドポイントの追加、`resolve` / `close` 相当の機能実現）。 |
| **patch** | 挙動を変えないか、バグ修正・内部改善のみ（`fix` / `hotfix` / `refactor` / ドキュメント・テストのみ等）。 |

次バージョンの算出:

- 見つかった最大の種別に応じて major / minor / patch を 1 つ上げ、下位は 0 に戻す（例: minor なら `1.4.2 → 1.5.0`）。
- **0.x.y（前バージョンが `1.0.0` 未満）の特例** → まだ不安定版なので、**破壊的変更があっても major は上げず minor を上げる**（例: `0.4.1 → 0.5.0`）。この場合は「0.x のため破壊的変更でも minor 扱い」である旨を確認時に明示する。
- ユーザーが種別やバージョンを明示している場合（「メジャーで」「1.0.0 にして」等）はそれを優先する。
- プレリリースを求められた場合（「beta で」等）は `1.0.0-beta.0` のような識別子を付ける。

判断は確認時に**根拠とともに**提示します（後述）。

## 5. リリースノートの生成

差分から、利用者目線のリリースノートを合成します。中間コミットの過程（"fix typo", "WIP", "address review" 等）は含めず、最終的な変更だけをまとめます。

```markdown
## <next>

### Added
- 〜を追加

### Changed
- 〜の挙動を変更

### Fixed
- 〜を修正
```

- 該当する変化が無いセクションは省く。破壊的変更がある場合は **Breaking changes** を最上部に立て、移行方法も書く。
- リポジトリに `CHANGELOG.md` があれば、その既存の書式・見出しスタイルに合わせる。更新まで行うかはユーザーに確認する（CHANGELOG への追記はコミットが必要になるため）。

## 6. リリース内容の確認

タグ作成・Release は**取り消しづらい外向きの操作**（タグは共有され、Release は関係者に通知される）なので、実行前に必ずユーザーに内容を見せて了承を取ります。

提示する内容:

- `<prev>` → `<next>`（と bump 種別）
- **バージョンをそう判断した根拠**（例: 「公開関数 `foo` のシグネチャを変更 → 破壊的だが 0.x のため minor」）
- リリースノート本文
- 何を作るか（注釈付きタグのみ / タグ + GitHub Release）

ユーザーが了承したら次へ進みます。修正指示があれば反映してから再提示します。

## 7. 注釈付きタグの作成

git-bump 系のツールは既定で**注釈付き（annotated）タグ**を読むため、軽量タグではなく注釈付きで作成します。

```
git tag -a <next> -m "<next>" -m "<リリースノート本文>"
```

- 第 1 の `-m` はタグ名（要約）、第 2 の `-m` 以降に本文。本文が長い場合はファイルから渡してもよい（`git tag -a <next> -F notes.md`）。
- 手順 5 で `CHANGELOG.md` を更新しコミットした場合は、**その更新を含むコミット**にタグを打つ（古い HEAD に打つと CHANGELOG エントリがリリースに含まれない）。
- タグはローカルに作られるだけで、push するまでリモートには出ません。

## 8. タグの push

タグはローカルに作られるだけなので、リモートへ反映するには push します。**まず一度 `git push origin <next>` を試します**（push 先は明示する）。

- **成功した場合** → タグが origin に反映される。
- **失敗した場合**（認証・パスフレーズ入力が対話的に必要、権限が無い等）→ 無理に繰り返さず、その時点でユーザーに push を依頼する。失敗の内容を簡潔に伝え、実行してほしいコマンド（`git push origin <next>`）を示す。ユーザー自身のシェルなら対話的な入力もできる。

GitHub Release を作る場合は、タグが origin に push 済みであることをこの後の前提にします。タグ作成だけで十分なら、ここで完了です。

## 9. GitHub Release の作成（任意）

GitHub リポジトリで、ユーザーが GitHub Release も希望する場合のみ行います。タグが push 済みであることを確認してから:

```
gh release create <next> --title "<next>" --notes-file notes.md
```

- ノート本文はコマンドラインに直書きせず、ファイルに書き出して `--notes-file` で渡す。
- プレリリースの場合は `--prerelease` を付ける。
- 最新版でない（過去バージョンの後追い等）場合は `--latest=false` を検討する。
- `--draft` はユーザーが明示した場合のみ。

## 10. 後処理

- 作成したタグ名と、GitHub Release を作った場合はその URL をユーザーに伝える。
- まだタグを push していない場合は、push 方法（手順 8）を改めて案内する。
- 一時的に作ったノートファイル（`notes.md` 等）は片付ける。

## フォールバック: GitHub MCP

`gh` が使えず GitHub MCP が利用できる環境では、GitHub MCP の Release 作成ツールで同じことを行います。その場合も「差分からバージョンとノートを合成」「注釈付きタグを作る」「push はまず試し、失敗したらユーザーに依頼」「作成前の確認」という方針は変わりません。
