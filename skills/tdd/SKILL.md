# TDD (テスト駆動開発)

Kent Beck の "Canon TDD" の定義に基づいて開発を進めるためのスキルです。動いているコードを壊さずに、新しい振る舞いを少しずつ・確信を持って育てていくための規律を提供します。

## TDD が解決すること

TDD は「システムの振る舞いを変更する」ためのプログラミングのワークフローです。ゴールは次の状態を**同時に**満たすこと:

- 既存の動いている機能がすべて動いたままであること
- 新しく望む振る舞いが期待どおりに動くこと
- システムが今後の変更を受け入れられる設計になっていること
- そして、プログラマ（とチーム)が上の 3 つに**確信**を持てること

最後の「確信」が肝です。TDD はテストを書く手法であると同時に、変更を小さく刻み、各ステップでフィードバックを得ることで「いま自分は正しい場所に立っている」と確信しながら進むための手段です。

TDD をやるかどうかは個人の選択ですが、自分の仕事に責任を持つなら、その質にも責任を持つ ── というのが Kent Beck の立場です。

## 核心: 設計を2種類に分けて、同時にやらない

TDD を理解する鍵は、**設計には2種類ある**と知ることです。

- **インターフェイス設計**: その振る舞いを「どう呼び出すか」。関数のシグネチャ、入出力、名前。
- **実装設計**: その振る舞いを「どう実現するか」。内部のアルゴリズム、データ構造、抽象化。

TDD はこの2つを別々のタイミングで考えさせます。テストを書くとき(Red)は主にインターフェイスに向き合い、実装の細部は脇に置きます。実装の作り込みは Refactor まで遅らせます。これが**「帽子の被り替え」**です ── 「動かす」帽子と「きれいにする」帽子を同時にかぶらない。一度に1つのことだけを考える方が、脳にとっても優しく、ミスも減ります。

「動かしてから、正しくする (Make it work, then make it right)」の順序を崩さないこと。

## ワークフロー: List → Red → Green → Refactor

よく "Red-Green-Refactor" と呼ばれますが、正確には先頭に**テストリスト**のステップがあります。**リスト → レッド → グリーン → リファクタ**。

### ステップ 0: テストリストを作る

これから実装する振る舞いについて、**期待される振る舞いを網羅的に列挙**します。

- 正常系（代表的なケース)
- エラーケース、例外的な入力
- 境界値・エッジケース（空、0、最大、null、重複 など）

これは「振る舞いの分析」であって、実装の設計ではありません。**ここで実装の都合を混ぜ込まないこと。** 「内部でこのクラスを作って…」のような実装判断はこの段階では不要で、後で(Refactor で)いくらでも考えられます。リストは外から見た振る舞いに集中させます。

リストは最初から完璧でなくてよく、開発中に新しいケースに気づいたら追加し、片付いたら消していく**生きたメモ**として扱います。

### ステップ 1 (Red): リストから1つ選び、失敗するテストを書く

テストリストから**ちょうど1つだけ**選び、それを「準備 (Arrange) → 実行 (Act) → 検証 (Assert)」を含む自動テストとして書きます。

そして**テストを実行し、失敗することを確認します。** これは省略してはいけません。失敗を見ずに進むと、そのテストが本当に対象の振る舞いを検証しているのか分からないままになります（書いた瞬間に通ってしまうテストは、何も検証していない可能性がある)。

どのテストから書くかの選択は、軽視されがちですが**重要なスキル**です。Kent Beck は、テストを選ぶ順番が「プログラミングの快適さと最終的な成果の両方に重大な影響を及ぼす」とし、そしてこの勘所は**経験を通じてのみ身につく**と述べています。明確なルールがあるわけではありません。

これは人間のプログラマを念頭に置いた言明ですが、生成 AI(あなた)の場合は、モデルが賢くなるにつれてより良い順序を選べるようになることが期待できます。下の目安に縛られすぎず、**文脈を踏まえたそのときのあなた自身の判断**で、次に書くべきテストを選んで構いません。

経験を補う実践的な目安（規則ではなくヒント):

- 最初は**単純で代表的なケース**から。早く緑にして足場を作る。
- 次の一歩が**小さくなる**ものを選ぶ。1つ前で書いたコードを少しだけ拡張すれば通るもの。
- 大きく一気に作らないと通らないテストは後回しにする。

迷ったら一番簡単なものから。

### ステップ 2 (Green): テストを通す

プロダクトコードに変更を加え、**いま書いたテストと、既存のすべてのテスト**が通る状態にします。

ここでの唯一の目的は「緑にすること」です。きれいさは問いません。素朴な実装、ベタ書き、重複 ── 何でも構わないので、まず通す。設計の作り込みは次のステップまで我慢します（帽子を被り替えない)。

### ステップ 3 (Optionally Refactor): 必要に応じて実装を整える

このステップは**任意**です。原文も "Optionally Refactor"(必要に応じてリファクタリングを行う)であり、整えるべきものがなければ何もせず次のサイクルへ進んで構いません。

整える場合は、テストが緑になった**後で**、ようやく実装設計を改善します。テストという安全網がある状態で、振る舞いを変えずに内部構造をきれいにします。

- 重複を取り除く、名前を整える、構造を見直す。
- ただし**重複は「指令」ではなく「ヒント」**。重複を見つけたら「より良い設計が隠れているかも」と気づくきっかけにする、という意味であって、機械的にすべての重複を消す義務はありません。
- **必要以上にリファクタリングしない。** 今のテストが要求していない抽象化を先回りして作り込まない（早すぎる抽象化)。設計は今分かっていることに対して最小限に。

### ステップ 4: 繰り返す

テストリストが空になるまで、ステップ 1 (Red) に戻ります。1サイクルで片付いた項目をリストから消し、途中で気づいた新しいケースを追加します。リストが空になったら、その振る舞いの変更は完了です。

## よくある間違い

各ステップで規律が崩れる典型パターン。サイクルが回らなくなったらここを疑う。

**テストリストの段階で:**
- 実装の設計判断（内部クラス構成など)をリストに混ぜ込む → リストは外から見た振る舞いだけにする。

**Red(テストを書く)で:**
- テストリストの全項目を**先に**まとめてテストコードへ翻訳してしまう → 1度に書くのは1つだけ。まとめ書きすると緑にするステップが巨大化し、フィードバックが遅れる。
- アサーション(検証)のないテストを書く → 何も確認していない。
- **失敗を確認せずに**先へ進む → そのテストが効いている保証がない。

**Green(通す)で:**
- アサーションを消したり弱めたりして「通ったことにする」 → 自己欺瞞。
- テストを実行して出た**実際の値**をそのまま期待値にコピペする → テストとコードの二重チェックという価値が失われる。期待値は「こうあるべき」という独立した根拠から書く。
- このタイミングでリファクタリングを混ぜ込む → 帽子の被り替え。動かすことだけに集中する。

**Refactor(整える)で:**
- テストが緑になる**前に**リファクタリングを始める → 安全網がない状態で構造をいじることになる。
- 必要以上のリファクタリング・早すぎる抽象化 → 今のテストが求めていない複雑さを持ち込む。

## 進め方のまとめ

1. **テストリスト**を作る（振る舞いを列挙、実装は混ぜない)。
2. リストから**1つ**選び、**失敗する**テストを書く (Red)。失敗を必ず目で確認する。
3. 最短で**緑**にする (Green)。きれいさは問わない。
4. 必要なら緑のまま**整える** (Optionally Refactor)。整えるものがなければ飛ばしてよい。重複はヒント、やりすぎない。
5. 片付いた項目を消し、気づいた項目を足し、リストが空になるまで 2 へ戻る。

小さく刻み、各ステップでフィードバックを得て、「動かす」と「きれいにする」を分けて、確信を積み上げながら進むこと。これが TDD です。
